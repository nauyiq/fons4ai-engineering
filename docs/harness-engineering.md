# Fons4AI Harness Engineering 方案

## 1. 目标

Fons4AI Harness Engineering 的目标是建立一套可迁移、可验证、可演进的 Agent 驾驭体系，使 Agent 在企业软件项目中能够稳定执行以下工作：

- 理解任务意图和边界。
- 按项目事实加载最小必要上下文。
- 遵守团队工程规则和技能职责边界。
- 通过测试、构建、脚本或审查清单证明完成。
- 将失败经验按归因沉淀到项目本地或上游技能仓库。

这套体系不是为了让流程变重，而是为了让 Agent 的行为可控、结果可验、经验可复用。

## 2. 定位边界

当前定位：

> Fons4AI Engineering 是面向企业软件交付的 Harness Engineering 实践。

长期演进：

> 在 Harness Engineering 成熟后，逐步演进为具备反馈回流、自我评估和持续优化能力的 Loop Engineering 体系。

二者关系如下：

| 概念 | 解决的问题 | 当前状态 |
| --- | --- | --- |
| Prompt Engineering | 如何写好单次提示 | 作为基础能力存在 |
| Harness Engineering | 如何驾驭 Agent 按工程规则交付 | 当前主线 |
| Loop Engineering | 如何让执行反馈持续改进体系 | 后续演进目标 |

## 3. 体系分层

### 3.1 Intent Harness

解决“用户到底要什么”。

核心产物：

- 需求说明书
- 变更请求 CR
- 验收标准 AC
- 阻塞问题与澄清记录

关键规则：

- 不把模糊表达当作需求确认。
- 不把 Agent 推断写成用户已确认事实。
- 需求进入实现前必须有范围、验收和完成定义。

### 3.2 Context Harness

解决“Agent 应该读什么，不该读什么”。

核心产物：

- `AGENTS.md`
- `.specify/memory/index.md`
- 项目级知识基线
- 领域或技术能力域文档
- 知识卡片
- 任务中的 `Files:` 范围

关键规则：

- 入口文档负责导航，不承载长篇知识。
- 按任务目标加载最小必要上下文。
- 知识事实必须标注来源，区分用户确认、正式文档、源码事实和推断。

### 3.3 Skill Harness

解决“Agent 以什么角色、按什么契约工作”。

核心产物：

- `skills/*/SKILL.md`
- 技能输入、前置条件、输出、完成标准
- 技能间 handoff 信息

关键规则：

- 每个技能只负责自己的职责边界。
- 实现类技能必须有最新用户授权。
- 规划类技能不得直接写业务代码。
- 知识类技能只沉淀已验证事实。

### 3.4 Rule Harness

解决“团队工程规则如何被稳定执行”。

核心产物：

- `.specify/rules/agent运行规则.md`
- `.specify/rules/代码编写规范.md`
- `.specify/rules/sdd团队协作规范.md`
- 规则校验脚本

关键规则：

- 规则必须短、明确、可执行。
- 项目事实不应堆进规则文件。
- 能用脚本、lint、测试或 CI 校验的约束，不只依赖文字提醒。

### 3.5 Feedback Harness

解决“怎么证明 Agent 做对了，失败后怎么归因”。

核心产物：

- 测试结果
- 构建结果
- 实施报告
- BUG 修复报告
- 失败报告
- DDL 或数据变更审查记录

关键规则：

- 任务完成必须绑定验证证据。
- 验证失败时保留任务未完成状态。
- 失败要区分项目私有原因和通用技能原因。

### 3.6 Learning Harness

解决“经验如何从项目中抽象成可复用资产”。

核心产物：

- 失败案例库
- 上游反馈单
- 回放样例
- 技能或模板修订记录
- 校验器改进记录

关键规则：

- 项目私有问题留在项目本地。
- 通用技能、模板或校验能力问题反馈回本仓库。
- 反馈必须经过脱敏、归因和抽象，不直接回流业务敏感信息。

## 4. 项目内闭环与上游闭环

业务项目接入 Fons4AI 技能后，失败处理分两层。

### 4.1 项目内闭环

适用于项目私有问题，例如：

- 业务术语不清。
- 项目环境缺依赖。
- 本项目规则缺失。
- 某个接口或字段口径未沉淀。

处理方式：

- 更新项目 `AGENTS.md`。
- 更新项目 `.specify/memory/`。
- 更新项目 `.specify/rules/`。
- 补充测试、环境说明或本地任务模板。

### 4.2 上游闭环

适用于可复用的通用问题，例如：

- 技能授权判断不清。
- 任务模板经常遗漏验证项。
- 多个项目重复出现同类误判。
- 校验脚本无法发现明显产物错误。
- 高风险任务完成门禁不足。

处理方式：

- 在业务项目生成失败报告。
- 抽象出不含敏感信息的上游反馈单。
- 在本仓库修改技能、模板、规则或校验脚本。
- 用回放样例验证修改是否解决同类问题。

## 5. SDD 与 Harness 的关系

SDD 是当前 Harness 的主要执行骨架。

它负责把需求、设计、任务和验证串联起来：

```text
需求说明书
  -> 技术设计说明书
      -> 任务规划
          -> 实现报告
              -> 知识汇总
```

但 Harness 不等于 SDD。Harness 还包括：

- 项目规则。
- 上下文加载策略。
- 技能路由。
- 环境准备。
- BUG 修复流程。
- 失败反馈与上游改进。

因此，后续优化应避免把所有任务都强制塞进完整 SDD，而是按风险和影响范围选择处理强度。

## 6. 第一阶段落地范围

第一阶段只做概念归一，不改动现有技能行为。

完成标准：

- 仓库顶层有明确 Harness Engineering 定位。
- 文档说明 Harness 与 Loop 的关系。
- 文档说明 SDD 是 Harness 的执行骨架之一。
- 文档明确项目内闭环和上游闭环的边界。
- 后续结构升级和反馈闭环有稳定术语基础。

暂不做：

- 不重写现有技能。
- 不调整 skill 路由门禁。
- 不新增跨项目反馈自动化。
- 不引入新的发布或安装机制。

## 7. 后续路线

阶段 2：结构升级。

- 统一技能契约结构。
- 补齐风险分层门禁。
- 整理模板、规则和校验脚本。
- 建立任务类型到验证方式的映射。

阶段 3：反馈闭环。

- 增加失败报告模板。
- 增加上游反馈单模板。
- 建立可脱敏的失败归因分类。
- 建立典型任务回放样例。
- 逐步引入自动评估和回放验证。
