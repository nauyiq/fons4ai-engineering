# Loop Phase 1

Loop Phase 1 的目标是把当前人工驱动的反馈闭环标准化，使每次上游改进都能追溯到反馈事实、修改清单、验证结果和全局同步状态。

本阶段不做自动回放引擎，不要求 Agent 自主长期循环。Loop 只能在有新证据、新用户确认或新产物时继续推进。

## 1. 阶段定位

Loop Phase 1 解决的问题：

- 反馈单已经存在，但后续如何转成修改清单不稳定。
- 技能、模板或校验器修改后，缺少统一的验证与同步记录。
- 用户在闭环中的决策点需要被显式记录。

Phase 1 的产物是人工可审查的闭环记录，不是自动评估系统。

## 2. 闭环路径

```text
业务项目试点反馈
  -> 上游反馈单
      -> Loop 改进记录
          -> 修改技能/模板/校验器/文档
              -> 仓库校验
                  -> 全局 skills 同步
                      -> 关闭或进入待观察
```

## 3. 用户职责

用户在 Phase 1 中负责高价值决策：

- 判断反馈是否值得进入上游改进。
- 裁决项目私有问题和通用 Harness 问题的边界。
- 确认是否允许修改技能、模板、校验器或全局 skills。
- 决定是否接受暂缓项、外部 Gate 或待观察状态。
- 判断改进结果是否满足当前试点目标。

Agent 负责机械和证据工作：

- 读取反馈单和相关技能/模板/校验器。
- 归因问题并生成可修改清单。
- 执行授权范围内的文件修改。
- 运行校验并记录结果。
- 同步全局 skills 后做哈希比对。

## 4. Loop 改进记录

每次通用上游改进应生成或更新：

```text
spec/reports/loop-improvements/<yyyymmdd>-<问题简述>-loop改进记录.md
```

使用模板：

```text
templates/loop-improvement-record-template.md
```

改进记录至少包含：

- 来源反馈单。
- 问题分类。
- 用户决策。
- 修改清单。
- 验证命令和结果。
- 全局同步状态。
- 关闭条件。

## 5. 状态规则

| 状态 | 含义 |
| --- | --- |
| Draft | 证据不足、用户决策未完成或修改范围未确认 |
| In Progress | 已确认进入上游改进，正在修改或验证 |
| Verified | 修改完成且仓库校验通过 |
| Synced | 已同步到全局 skills 并完成一致性复核 |
| Deferred | 外部条件、证据或用户决策不足，暂缓 |
| Rejected | 判断为项目私有问题或不应进入上游 |

## 6. Phase 1 完成定义

单次 Loop 改进只有同时满足以下条件，才能视为关闭：

- 来源反馈单可定位。
- 修改清单已执行或明确暂缓。
- 仓库级校验已运行并记录结果。
- 若涉及技能包，全局 skills 同步状态已记录。
- 剩余风险和待观察项已写明。

## 7. 非目标

本阶段不做：

- 自动生成跨项目评分。
- 自动选择要修改的技能。
- 自动执行生产 DDL/DML 或外部系统写入。
- 自动创建长期知识事实。
- 无界重试或无人值守循环。
