## 现象

Agent 在执行 `admin-control-plane` 的 S2 风险门禁 T022 时，已经完成 admin-service 全量自动化回归，并通过只读 MCP 确认 MySQL、Nacos、Redis 的部分可读能力，但仍无法关闭跨服务 L3 验证风险。

具体表现：

- admin-service 全量测试通过，记录为 `Tests run: 73, Failures: 0, Errors: 0, Skipped: 0`。
- MySQL 只读验证确认 `sys_admin` 中 10 张 admin 目标表存在。
- Nacos 可读取 namespace 和网关动态路由配置。
- Redis 连接与认证可用，但未证明运行期已有治理数据。
- Nacos 服务实例查询对认证服务、网关服务相关服务名返回 404，无法证明服务发现实例列表。
- auth-service 真实 Dubbo RPC 登录、刷新、吊销 Token 与 OAuth Client 管理未完成真实联调，只能由 Mock/契约测试覆盖。
- Actuator 只读探测缺少可访问服务实例地址，只能由适配器级测试覆盖成功、不可用和白名单边界。
- 为避免污染共享运行配置，未对真实 Nacos 发布路由，也未对真实 Redis 写入黑白名单。

用户进一步提出疑问：仓库内本身包含认证服务和网关服务代码，Agent 是否应该自行启动这些服务完成验证；如果不能，是否说明当前项目缺少可持续验证环境或 CI/CD 编排契约。

## 期望行为

在 SDD 实现流程遇到跨服务 L3 验证缺口时，Harness 不应只停留在“缺少外部环境证据”或反复等待用户输入“继续下一步”。同类场景下建议具备以下处理能力：

- 识别当前任务是否属于跨服务集成验证，而不是普通单模块测试。
- 检查仓库内是否存在可启动的相关服务模块，例如 admin、auth、gateway。
- 检查是否存在可持续验证契约，例如 local-integration profile、启动脚本、测试 namespace、测试 Redis key prefix、测试数据库 schema、健康检查脚本、服务清理脚本、CI job 或 compose 编排。
- 如果存在验证契约，在用户授权后启动相关服务并采集 L3 证据。
- 如果不存在验证契约，明确输出项目侧缺口清单，并把任务保持未完成或标记为待用户暂缓确认。
- 对共享 Nacos、Redis、MySQL 等外部资源，要求隔离 namespace、group、schema、key prefix 和清理策略后才能执行写入验证。
- 在实施报告和风险清单中区分 Mock/契约测试证据、只读环境证据、真实跨服务运行证据，避免把低等级证据误写为发布就绪。

## 初步归因

- 归因结论：待观察
- 归因说明：
  - 项目侧确实缺少可直接由 Agent 复用的本地/测试集成验证编排，因此 `ENV_LOCAL` 成立。
  - T022 不关闭是正确行为，因为真实服务发现、真实 auth RPC、真实 Actuator 探测和真实配置写入缺少 L3 Gate Evidence。
  - 但从 Harness 试点角度看，`fons4ai-sdd-implement` 与 `fons4ai-agent-env-readiness` 之间缺少明确 handoff：当仓库有相关服务代码但没有可持续验证环境时，应自动产出“集成验证环境契约缺口”，而不是只把任务卡在未关闭状态。
  - 因此该问题同时暴露 `EVIDENCE_GAP` 和 `SKILL_CONTRACT`：Evidence 要求本身合理，但缺少如何发现、编排或声明环境能力边界的标准流程。
- 不确定项：
  - 该问题是否已在其他试点项目重复出现，当前未收集到跨项目证据。
  - 当前仓库是否已有未被 Agent 识别的启动脚本或 CI job，需要后续单独核查。
  - 是否允许 Agent 在共享 Nacos/Redis/MySQL 环境中自行注册临时服务或写入治理配置，需要项目侧明确授权和隔离策略。

## 是否建议回流上游

- 建议回流：是
- 判断依据：
  - 该问题不只是单个服务不可用，而是 SDD 实现、环境准备度、Evidence 门禁之间的协作契约不够明确。
  - 反馈目标不是要求 Harness 替项目凭空提供环境，而是要求 Harness 能识别“仓库有服务代码但缺少可持续验证编排”的状态，并产出标准化后续动作。
  - T022 的阻塞有明确实施报告和风险清单证据，可作为上游改进输入。
  - 跨项目重复性目前待观察，因此不建议直接定性为 `CROSS_PROJECT_REPEAT`。
- 若不回流，应留在业务项目的处理位置：
  - 新增 admin 集成验证环境任务。
  - 补充本地或测试环境启动脚本、隔离资源和清理策略。
  - 后续由用户确认是否允许 Agent 启动 auth、gateway、admin 进行联调验证。

## 建议修改位置

- skill：
  - `fons4ai-sdd-implement`：在 L3 验证缺失时增加“可持续集成验证环境发现”步骤；如果仓库内存在相关服务代码，先判断是否可启动、是否有隔离资源、是否有清理策略，再决定继续验证、阻塞或暂缓。
  - `fons4ai-agent-env-readiness`：增强跨服务任务的环境准备度清单，覆盖服务启动、注册发现、RPC、Actuator、配置中心写入、Redis 写入、数据库写入、清理策略和 CI 复用入口。
  - `fons4ai-harness-feedback`：可保留现状，本次反馈单模板能够承载问题。
- template：
  - 可考虑在 SDD 实施报告或 S2 风险清单模板中增加“集成验证环境契约”小节，明确服务启动方式、外部依赖、隔离资源、写入权限、清理策略、可重复命令。
- script：
  - 可选新增环境探测脚本模板，例如检查 Maven 模块、Spring Boot 启动入口、profile、Nacos namespace、服务注册、Actuator 端点和端口占用。
- docs：
  - 建议在 Harness 工程文档中补充“跨服务 L3 Evidence 的处理规范”：缺少环境时不关闭任务，但要输出项目侧缺口；仓库内有服务代码时不等于可自动联调。
- validator：
  - 可选增加验证报告校验项：如果任务声明发布就绪但存在未验证的真实服务发现、RPC 或 Actuator 链路，应提示证据不足。
