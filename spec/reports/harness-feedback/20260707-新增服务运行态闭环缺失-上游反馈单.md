## 现象

Agent 在实现任务中完成了业务组件、REST Controller、RPC 客户端、治理适配器和自动化单元测试，T021 集成回归任务也记录了回归通过和部分外部只读验证。但在 T022 发布门禁复检时才发现：新增服务缺少完整运行态闭环证据，无法证明该服务可以独立启动、注册到服务发现、通过自身 REST API 调用核心链路，并完成真实 RPC 联调。

该问题不应被视为单个项目的启动类遗漏，而是“新增可运行服务”在 SDD 设计和任务规划阶段缺少标准建模与强校验。

## 期望行为

Agent 或技能体系在同类场景下应该前置识别并处理：

- 当技术设计声明某模块提供 REST API、RPC Consumer、注册中心接入、Actuator 探测、配置治理或对外服务能力时，必须先判定模块形态。
- 若模块形态为独立运行服务，设计阶段必须生成运行态设计：启动入口、服务名、端口、注册发现、配置来源、健康检查、部署方式、最小运行验证。
- 任务规划阶段必须生成可执行任务：创建启动类、补齐运行配置、启动服务、注册检查、健康检查、至少一个核心 API 或 RPC 链路验证。
- S2 任务规划必须包含 Evidence Matrix，并明确哪些证据可自动验证、哪些需要隔离环境、哪些允许用户暂缓确认。
- 实现阶段不得仅凭 Controller/Service/Adapter 单元测试通过就声明服务级闭环。
- 若缺少运行态任务，`fons4ai-sdd-implement` 应停止并建议回到 `fons4ai-sdd-tasks` 补齐任务，而不是到最终 T022 才暴露。

## 初步归因

- 归因结论：通用上游问题
- 归因说明：该问题来自 SDD 契约和模板没有把“新增服务是否可独立运行”作为设计必填项，也没有在任务规划和校验器中对 REST/RPC/注册中心特征做反向约束。它具有跨项目通用性，适用于新增后端服务、控制面服务、基础设施服务、聚合服务、网关插件服务等场景。
- 不确定项：是否已有上游 validator 能覆盖该规则但未在当前项目启用，需在 `fons4ai-engineering` 中确认。

## 是否建议回流上游

- 建议回流：是
- 判断依据：问题不依赖具体业务字段或项目专有实现；触发条件可抽象为“新增或改造可运行服务模块”；修复方式可落在 SDD 模板、任务规划模板、Evidence Gate 和校验器中。
- 若不回流，应留在业务项目的处理位置：可在项目 `.specify/rules/` 中临时增加“新增服务运行闭环规则”，但这只能覆盖当前项目，不能解决通用 SDD 产物质量问题。

## 建议修改位置

- skill：
  - `fons4ai-sdd-design`：新增“运行态设计”必填判断。
  - `fons4ai-sdd-tasks`：新增“可运行服务闭环任务”生成规则。
  - `fons4ai-sdd-implement`：在执行前检查运行态任务是否存在；缺失时停止并要求回到任务规划。
- template：
  - 技术设计模板增加“服务形态与运行态设计”章节。
  - 任务规划模板增加“新增服务运行闭环任务组”。
  - S2 风险清单模板增加 Evidence Matrix。
- script：
  - SDD artifact validator 增加服务形态规则。
  - 任务规划 validator 增加 REST/RPC/注册中心特征检测。
- docs：
  - 增加“新增可运行服务的 SDD 最小闭环”说明。
- validator：
  - 检测 `interfaces/rest/**Controller.java`、`@DubboReference`、`@FeignClient`、`@RabbitListener`、`@KafkaListener`、`spring.application.name`、注册中心依赖等运行态特征。
  - 若存在运行态特征，校验是否存在启动入口、运行配置、启动验证任务、注册/健康/RPC 或 API Evidence。

## 建议校验规则

1. 如果新增模块命名符合 `*-service`，且包含 REST Controller，则技术设计必须声明服务形态。
2. 如果模块声明为独立运行服务，则任务规划必须包含启动类、运行配置、启动验证、健康检查和服务注册验证。
3. 如果存在 `@DubboReference`、`@FeignClient` 或其他远程调用 Consumer，则任务规划必须包含真实 Provider 可达验证，或用户确认的暂缓说明。
4. 如果存在配置中心引用、classpath 配置引用或外部 dataId，则任务规划必须验证引用存在、注入方式明确或暂缓说明明确。
5. 如果 SDD 等级为 S2，T021/T022 不得仅以单元测试和 Mock 测试通过作为跨服务完成证据。
6. 如果外部写入涉及共享 Nacos、Redis、数据库、MQ 或配置中心，必须走隔离环境验证、只读降级验证加暂缓确认，或明确阻塞。
7. 如果最终交付声明包含“可发布”或“门禁关闭”，实施报告必须具备编译、单测、启动、注册、健康、核心 API/RPC、数据结构、外部依赖和人工 Gate 的 Evidence Matrix。
