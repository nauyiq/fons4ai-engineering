## 现象

Agent 实际哪里卡住、误判、遗漏、过度执行或无法验证：

- 正式 SDD 文档正文发生变更后，文档头部的更新时间没有同步更新；校验阶段没有发现该问题，导致用户在复盘时发现“正文已变更，但更新时间仍停留在旧日期”。
- 增量实现涉及可视化管理页面时，Agent 直接进入页面代码实现和功能验证，未先输出页面信息架构、低保真结构、布局方案、关键交互流或视觉验收标准，也未设置用户确认 Gate。
- 页面最终虽然具备运行、打包和 API 调用能力，但视觉形态偏原始；用户需要在实现后反向指出“只想开页面时没有看到页面设计”，说明 SDD 实现流程对 UI/页面类任务缺少明确的设计前置检查。
- 现有 Evidence 更偏向“可运行、可打包、测试通过”，不足以覆盖“页面是否经过设计确认、是否满足控制台使用体验”的质量证据。

## 期望行为

Agent 或技能体系在同类场景下应该如何处理：

- 只要正式 SDD 需求、设计、任务规划或实施报告发生正文修改，技能应要求同步文档头部更新时间，并补充版本记录或变更记录。
- 校验器应能发现“文档正文有本次日期或新增变更内容，但头部更新时间仍为旧日期”的明显不一致。
- 当任务涉及页面、控制台、前端、可视化界面、模板引擎页面、管理后台等交互型交付物时，实现前应插入 UI 设计确认 Gate。
- UI 设计确认 Gate 至少应要求输出页面信息架构、页面清单、主要布局、关键操作流、状态反馈、敏感信息展示策略和验收口径。
- 用户确认设计方案后再进入编码；若用户明确要求快速实现，也应在实施报告中记录“用户跳过 UI 设计确认”的决策证据。

## 初步归因

- 归因结论：通用上游问题候选
- 归因说明：
  - 更新时间未同步属于 SDD 文档元数据治理缺口。当前技能流程强调正文、任务状态和验证证据，但没有把“修改正式文档必须同步元数据”做成强规则或校验项。
  - 页面未先设计属于 SDD 实现职责边界缺口。增量变更进入实现阶段后，Agent 对页面类任务采用了“功能闭环优先”的后端式实现路径，没有自动识别 UI 设计 Gate。
  - 这两个问题都可抽象为上游规则候选：一个是文档生命周期元数据一致性，一个是交互型交付物的设计确认门禁。
- 不确定项：
  - 当前仅基于一个试点项目反馈，跨项目重复性待观察。
  - 尚未确认现有模板中是否已有隐藏的 UI Gate 要求但未被执行；需要上游检查 `sdd-change`、`sdd-implement`、任务模板和校验器之间的职责分配。
- 证据约束：
  - 没有把本次问题写成多项目重复问题。
  - 没有把单次页面质量问题上升为所有前端任务的已验证缺陷。
  - 反馈单只作为上游改进输入，不代表批准直接变更上游技能库。

## 是否建议回流上游

- 建议回流：是
- 判断依据：
  - 文档更新时间与正文变更不一致属于通用 SDD 文档治理风险，适合沉淀为模板要求和校验器规则。
  - 页面类任务缺少设计确认 Gate 会影响任何采用 SDD 实现可视化界面的项目，适合沉淀为技能契约和任务规划要求。
  - 当前证据足以支撑“候选上游改进”，但跨项目重复性仍写为待观察。
- 若不回流，应留在业务项目的处理位置：
  - 可作为项目本地规则补充到 Agent 运行规则或 SDD 协作规范，但仅本地修补不能防止其他试点项目重复出现。
