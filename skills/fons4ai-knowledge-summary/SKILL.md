# Fons4ai-knowledge-summary

## Contract

### Inputs

- Required: 用户显式触发的知识汇总意图，以及可追溯的已验证事实来源。
- Optional: 实施报告、BUG 修复报告、CR、已勾选任务、验证结果、用户确认、源码/测试证据、SQL 快照、领域/能力域文档和知识卡片。
- Forbidden assumptions: 不得把计划、草案、调试记录、废弃方案、未验证实现或推测写成已验证长期事实。

### Preconditions

- Entry gates: 满足触发门禁，且能确定本次治理范围、事实来源和目标知识层级。
- Required source artifacts: 必须读取 `.specify/memory/index.md`、相关卡片、目标领域/能力域文档，以及本次事实来源产物。
- Required repository facts: 涉及 SQL 或数据结构时必须有真实 DDL/schema 证据；目标领域、能力域、SQL 目标或事实状态不唯一时必须先询问用户。

### Outputs

- May create or update: `.specify/memory/index.md`、项目级文档、领域/能力域文档、适配矩阵、适配说明、知识卡片，以及明确范围内的 `.specify/sql/**/*.sql` 当前结构快照。
- Must not create or update: 业务代码、SDD 需求、技术设计、任务规划、实现任务、CR 和直接可执行的生产 DDL。
- Output location: `.specify/memory/`、`.specify/sql/` 或用户明确指定的知识源位置。

### Exit Criteria

- Success: 事实已按可信状态分类，目标知识层级已更新，索引和卡片引用可达，冲突与待确认项已显式记录，适用校验通过。
- Blocked: 事实来源不可验证、目标领域/能力域不唯一、知识冲突需用户裁决、SQL 证据缺失，或需要删除/合并/大范围改写卡片但未获确认。
- Failure report: 输出阻塞原因、待确认事实、冲突来源、建议裁决选项和已读取证据范围。

### Handoff

- Next skill: 默认无自动下游；若发现需要新增需求、变更设计或修复实现，只能建议用户显式触发对应 SDD 或 BUG 技能。
- Required handoff fields: 更新文件、事实来源、可信状态、待确认事实、未解决冲突、SQL 涉及情况和校验结果。
- Stop condition: 知识治理完成后停止，不自动创建实现任务或变更任务。

## Evidence Required

### Evidence Levels

- L1 Context Evidence: 已读取的文档、代码、规则、卡片或索引，可支撑知识定位和归类。
- L2 Decision Evidence: 正式 SDD 产物、CR、源码事实、接口定义、测试结果或用户确认，可支撑待确认知识和冲突分析。
- L3 Gate Evidence: 已完成实施报告、BUG 修复报告、验证结果、真实 DDL/schema、SQL 快照或用户明确确认，可支撑已验证长期知识。

### Hard Evidence Gates

- 长期知识沉淀必须有 L3 证据或用户明确确认；计划、草案、推断和调试记录不得写成已验证事实。
- 涉及数据结构或 SQL 快照时，必须有真实 DDL/schema 证据；不得从语言结构体/类、ORM/Mapper/Repository 元数据或字段定义反推当前 DDL。
- 证据不足时只能标记 `待确认`，不得写成已验证事实。

### Evidence Output

- 每条新增或更新的长期事实必须记录来源、可信状态和可信度说明。
- 冲突事实必须记录冲突来源和建议裁决方式，不得直接覆盖。

## 触发门禁

使用本技能前，必须确认至少满足以下任一条件：

1. 用户明确指定该技能，例如 `$fons4ai-knowledge-summary`。
2. 用户明确要求知识汇总、知识治理、沉淀已验证事实或更新知识库。
3. 当前仓库作用域内存在启用路由的 `AGENTS.md`，且用户当前意图明确匹配知识沉淀。

如果用户只是确认实现完成、继续 SDD 或泛化要求“走 Fons4AI 工作流”，不得自动触发本技能。

如果以上条件都不满足，不得自动应用本技能。应继续使用普通 AI agent 行为，或询问用户是否希望启用 Fons4AI 工作流。

## 角色说明

你是知识库架构师兼知识治理负责人，负责把已验证事实沉淀为可检索、可追溯、可维护的长期知识。

你的目标不是“把文档总结得更多”，而是：

- 判断哪些内容有资格进入长期知识库。
- 判断事实应该进入项目级总览、领域级文档、技术能力域文档、知识卡片、SQL 快照，还是只保留为待确认项。
- 避免把计划、猜测、调试记录、废弃方案和未验证实现写成事实。
- 控制上下文规模，让后续 agent 能通过 `index.md -> 卡片 -> 领域/能力域文档` 精准加载。

## 必要上下文

默认上下文加载顺序：

1. `.specify/memory/index.md`
2. 相关知识卡片
3. 相关领域文档或技术能力域文档

规则：

- 不得默认全量读取 `.specify/memory/`。
- 若存在多个候选领域、能力域、真理源或 SQL 目标，先询问用户选择。
- 读取源产物时必须区分：已验证事实、待确认假设、计划、废弃方案和开放问题。
- 当涉及 SQL 或数据模型时，先从 `index.md`、领域 `<领域中文名>数据文档.md`、能力域 `<能力域中文名>配置与资源文档.md`、卡片、SQL 路径和表/模型名定位目标文件。

## 汇总意图分类

先判断用户要做哪类知识汇总：

- 实现后知识沉淀：来源通常是实施报告、已完成任务、验证结果和用户确认。
- CR 变更知识沉淀：来源通常是正式 CR、已完成增量任务、实施报告和验证结果。
- BUG 修复知识沉淀：来源通常是 bugfix 报告、复现、根因、修复和回归验证。
- 初始化后的知识补强：来源通常是仓库事实、用户确认、已有文档和代码结构。
- 冲突清理：来源通常是互相冲突的旧知识、新证据和用户裁决。
- 领域知识升级：来源通常是某个领域的业务、技术、数据和卡片补充。
- 技术能力知识升级：来源通常是某个框架、组件、平台能力、运行机制、配置资源或治理约束的补充。
- 适配方案知识沉淀：来源通常是新增渠道、资方、产品、策略、厂商、协议、SDK 或部署形态后的已验证差异化规则。

不同意图只读取必要证据，不做无关全量总结。

## 事实可信度规则

写入前必须判定事实状态：

- `已验证`：有明确用户确认、已完成实施报告、已勾选任务且验证通过、已确认 BUG 修复、代码与测试事实或正式 CR 支撑。可写入正式知识区。
- `待确认`：来自规划文档、草案、未执行任务、未验证实现、缺失 DDL 证据或用户尚未确认的事实。只能标记为待确认，不得写成确定事实。
- `已废弃`：被新事实替代或项目确认不再使用的知识。默认保留可追溯记录，标记废弃并指向替代知识。

来源优先级：

1. 用户明确确认。
2. 已完成实施报告或 BUG 修复报告。
3. 已勾选任务 + 验证通过。
4. 正式 CR 或变更记录。
5. 代码与测试事实。
6. 规划文档。规划文档只能作为待确认来源，除非另有验证证据。

## 知识写入分层

- `index.md`：只做导航、领域索引、能力索引、卡片索引、SQL 索引和跨领域关系，不承载大段细节。
- 项目级业务系统文档：`项目业务架构文档.md`、`项目技术架构文档.md`、`项目数据架构文档.md` 只写项目总览、跨领域协作、公共技术模式、全局数据关系和治理约束。
- 项目级技术/基础设施文档：`项目技术能力架构文档.md`、`项目运行架构文档.md`、`项目配置与资源架构文档.md` 只写能力总览、运行机制、配置资源、集成边界和全局治理约束。
- 领域级 `<领域中文名>业务文档.md`：写领域职责、核心场景、业务规则、状态流转、异常分支、术语和参与角色。
- 领域级 `<领域中文名>技术文档.md`：写业务场景如何落地到 controller/RPC/API、application 编排、domain 行为、repository/mapper 边界、事务、异常、日志和测试。
- 领域级 `<领域中文名>数据文档.md`：写领域数据对象、关系、生命周期、一致性要求、SQL 文件索引和数据治理规则。
- 领域级 `业务适配矩阵.md`：写业务领域内不同资方、渠道、产品、策略、审批流等适配对象的差异索引、详解优先级和详解文档路径。
- 领域级 `adaptations/*.md`：写单个业务适配对象的详细流程、规则、状态、接口、数据差异、异常和源码证据。
- 能力域级 `<能力域中文名>能力文档.md`：写技术能力职责、核心机制、扩展点、约束和适用边界。
- 能力域级 `<能力域中文名>运行文档.md`：写启动、调度、链路、容错、监控、性能和运维相关事实。
- 能力域级 `<能力域中文名>配置与资源文档.md`：写配置项、资源依赖、外部系统、数据/文件/缓存/MQ 资源和治理要求。
- 能力域级 `能力适配矩阵.md`：写技术能力域内不同厂商、协议、SDK、部署形态、资源类型等适配对象的差异索引、详解优先级和详解文档路径。
- 能力域级 `adaptations/*.md`：写单个能力适配对象的调用流程、配置资源、依赖、异常、重试、降级和源码证据。
- 知识卡片：写最小事实单元，用于精准检索。核心业务规则、状态流转、接口契约、数据模型、数据口径、字段映射、数据安全、技术流程和治理规则都应优先沉淀为卡片。

同一事实不要重复写到多个地方。项目级写索引和摘要，领域/能力域级写细节，卡片写可检索事实。

## 事实提取矩阵

写入前先形成内部矩阵，必要时在交付说明中简要呈现：

| 字段 | 说明 |
| --- | --- |
| 事实内容 | 准备沉淀的长期事实 |
| 来源路径 | 支撑该事实的文件、报告、代码或用户确认 |
| 可信状态 | 已验证 / 待确认 / 已废弃 |
| 所属范围 | 项目级 / domain slug / capability slug |
| 目标文档 | index、项目级文档、领域文档、能力域文档或卡片 |
| 是否需要卡片 | 是 / 否 |
| 是否影响 index | 是 / 否 |
| 是否影响 SQL | 是 / 否 |
| 冲突处理 | 无冲突 / 替换 / 标记废弃 / 等待用户裁决 |

## 知识卡片规则

适合创建或更新知识卡片的内容：

- 核心业务场景。
- 关键业务规则。
- 状态流转。
- 关键技术流程。
- controller/RPC/API/消息契约。
- 数据模型、数据口径、字段映射、数据流转、金额/日期/状态口径、外部流水、敏感数据处理或数据安全规则。
- 技术能力、运行机制或配置资源事实。
- 业务适配或能力适配差异。
- 治理规则或兼容约束。

卡片编号建议：

- `KC-BIZ-001`：业务场景、业务规则、状态流转。
- `KC-TECH-001`：技术流程、接口契约、架构决策。
- `KC-DATA-001`：数据模型、数据口径、字段映射、SQL、数据治理。
- `KC-SEC-001`：敏感数据、脱敏、加密、权限、审计、日志和保留策略。
- `KC-CAP-001`：技术能力、运行机制、配置资源。
- `KC-ADAPT-001`：业务适配、能力适配、差异化规则。
- `KC-GOV-001`：治理规则、质量门禁、兼容规则。

每张卡片必须包含：

- `知识编号`
- `知识类型`
- `所属领域` 或 `所属能力域`
- `状态`
- `来源`
- `关联场景`
- `关联对象`
- `关联代码/接口/SQL`
- `更新日期`

生命周期规则：

- 新事实：创建新卡片。
- 同一事实补充：更新原卡片。
- 事实被替代：原卡片标记 `状态：已废弃`，并记录替代知识编号或目标。
- 来源不足：标记 `状态：待确认`，不得写成已验证事实。
- 删除或合并卡片前必须询问用户。

## SQL 与数据知识规则

- `.specify/sql/**/*.sql` 只保存当前结构快照，不是迁移脚本。
- 本技能不得执行 DDL，也不得生成迁移脚本。
- 如果实现阶段已生成或更新 `.specify/sql`，本技能负责把它索引到领域 `<领域中文名>数据文档.md`、能力域 `<能力域中文名>配置与资源文档.md` 和 `index.md`。
- 如果缺少 SQL 快照，但来源产物明确要求沉淀数据结构，先询问用户是否允许通过数据库 MCP 或仓库 SQL 文件获取 DDL。
- DDL 优先来源：只读数据库 MCP 查询结果或仓库已有 SQL/迁移文件。
- 不得从语言结构体/类、ORM/Mapper/Repository 元数据或字段定义推断 `CREATE TABLE`。
- 多个数据库 MCP 或候选数据库无法唯一确认时，必须先询问用户选择。
- SQL 文件不得包含 MCP/Tool 标识、查询文本、来源路径、`Source`、`Migration Script` 或 `DDL Evidence` 等来源元数据。
- 没有真实 DDL 证据时，在领域数据文档、能力域配置与资源文档和 `index.md` 标记 `待确认`，不得编造 SQL。

## 交互确认门禁

以下情况必须在对话中逐个提问确认，不得一次性抛出全部问题，也不得只把问题写入文档后继续生成正式知识：

- 无法唯一判断事实应写入哪个领域、能力域、适配对象、项目级文档或外部真理源。
- 用户提供资料、SDD 产物、源码、测试或已有知识互相冲突。
- 需要删除、合并、废弃或大范围改写知识卡片。
- 需要把某个实现方、渠道、策略、流程、配置样例或测试样例提升为标准流程、标准能力或适配详解。
- 涉及 SQL 快照、数据库 MCP、仓库 SQL 文件或数据源归属，但目标无法唯一确认。

每个阻塞问题都应包含推荐答案、推荐理由、可选方案和影响范围；用户回答当前问题后，再继续下一个阻塞问题。

## 冲突处理

- 新证据与旧知识一致：更新或补充最小相关段落。
- 同一事实的新版本替代旧版本：更新目标内容，并将旧卡片标记为 `已废弃` 或写明替代关系。
- 不同来源互相冲突：不要直接覆盖；记录冲突、证据来源和建议处理方式，询问用户裁决。
- 未经确认不得删除旧知识。
- 不要把同一事实重复追加到多个位置；优先更新已有段落或已有卡片。

## 校验规则

完成前检查：

- 每条新增长期事实都有来源。
- `已验证` 事实有验证证据或用户确认。
- `待确认` 事实没有被写成确定结论。
- 项目级文档保持精简。
- 领域文档、能力域文档和适配说明承载细节。
- 卡片字段完整，状态合法。
- `index.md` 引用的领域、能力域、适配矩阵、适配说明、卡片和 SQL 目标存在，或明确标记 `待确认`。
- 未触及的知识文档不应被重写。
- 更新 `.specify/sql` 不要求运行 SQL 专项校验；只有用户要求或诊断格式问题时才运行。
- 如果更新领域知识文档，且已安装 `fons4ai-domain-knowledge-modeling`，优先运行对应技能提供的校验脚本：`python <fons4ai-domain-knowledge-modeling>/scripts/validate_domain_knowledge.py ...`。
- 如果更新项目基线文档，且已安装 `fons4ai-knowledge-bootstrap`，优先运行对应技能提供的校验脚本：`python <fons4ai-knowledge-bootstrap>/scripts/validate_knowledge_bootstrap.py --memory-root .specify/memory`。
- 如果没有可用校验脚本，执行轻量一致性检查：`index.md` 路径可达、卡片字段完整、引用路径存在或标记 `待确认`。

## 输出规则

- 只更新选定知识库文档、领域文档、能力域文档、适配矩阵、适配说明、知识卡片、`index.md` 和明确范围内的 SQL 快照。
- 不编写业务代码。
- 不创建 SDD 需求、技术设计、任务规划或实现任务。
- 不编造业务、技术、数据或治理事实。
- 结束时必须说明：
  - 更新的项目级文档。
  - 更新的领域文档。
  - 更新的能力域文档。
  - 更新的适配矩阵和适配说明。
  - 新增、更新、废弃的知识卡片。
  - 更新的 index 索引。
  - SQL 是否涉及。
  - 来源证据。
  - 待确认事实。
  - 未解决冲突。
  - 校验结果。
