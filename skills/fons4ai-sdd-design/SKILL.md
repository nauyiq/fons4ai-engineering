# Fons4AI-sdd-design

## Contract

### Inputs

- Required: 正式 `spec/features/<yyyymmdd>/<功能中文名>-需求说明书.md`。
- Optional: `AGENTS.md`、`.specify/rules/sdd团队协作规范.md`、相关知识卡片、领域/能力域文档、SQL/schema 证据、接口文档、代表性源码和测试。
- Forbidden assumptions: 不得把草案需求、未确认数据口径、实体/ORM/Mapper 字段或单个实现样例当作已确认设计事实；不得凭经验生成存量 DDL。

### Preconditions

- Entry gates: 满足触发条件之一，且需求说明书不是草案或阻塞状态。
- Required source artifacts: 需求说明书必须完整读取；如技术设计已存在，必须先读取并判断合并策略。
- Required repository facts: 数据结构变更必须定位现状结构证据或标记待确认；标准扩展场景必须优先查找已有工作流、规则或知识卡片。

### Outputs

- May create or update: `spec/features/<yyyymmdd>/<功能中文名>-技术设计说明书.md`；必要的 S2 设计补充文件如 `contracts/`。
- Must not create or update: 需求说明书、任务规划、业务代码、执行型 DDL、SQL 快照、知识库真理源。
- Output location: `spec/features/<yyyymmdd>/`。

### Exit Criteria

- Success: 技术设计覆盖全部 AC，记录 SDD 等级、模块/接口/数据/状态/风险/验证策略；数据结构变更已包含结构变更详设。
- Blocked: 需求语义、关键数据口径、字段映射、现状 DDL/schema 证据、迁移位置、回滚策略或公共契约仍会影响设计正确性。
- Failure report: 汇报阻塞点、需回到的上游技能、一个最高影响问题或需要用户提供的证据。

### Handoff

- Next skill: `fons4ai-sdd-tasks`。
- Required handoff fields: 技术设计路径、SDD 等级、AC 覆盖、受影响模块/文件、接口契约、数据结构详设、DDL/SQL 影响、风险和验证策略、知识影响。
- Stop condition: 设计阻塞、输入为草案、缺少关键证据或用户只要求设计阶段时停止，不拆任务。

## Evidence Required

### Evidence Levels

- L1 Context Evidence: 已读取的需求、规则、代码、配置或文档，可支撑普通说明和影响分析。
- L2 Decision Evidence: 用户确认、正式 SDD 产物、源码事实、接口定义或测试结果，可支撑设计决策、接口契约和任务拆解依据。
- L3 Gate Evidence: 真实 DDL/schema、`.specify/sql` 快照、只读数据库查询、仓库迁移文件或用户明确确认，可支撑数据结构设计和结构变更结论。

### Hard Evidence Gates

- 数据结构、DDL、字段、索引、约束和迁移设计必须有 L3 证据；不得从实体类、Mapper、ORM 注解或 Java 字段反推当前 DDL。
- 公共 API、RPC、消息、权限、安全和事务边界变化至少需要 L2 证据。
- 推断只能作为影响分析，不得单独作为 AC、DDL、公共契约或完成状态依据。

### Evidence Output

- 技术设计必须包含 `证据清单`，列出关键结论、证据来源、证据等级和状态。
- 证据不足时必须标记 `待确认` 或阻塞，不得写成已确认设计事实。

## 触发门禁

使用本技能前，必须确认至少满足以下任一条件：

1. 用户明确指定该技能，例如 `$fons4ai-sdd-design`。
2. 用户明确要求进行技术设计补充、生成技术设计说明书或只处理设计阶段。
3. `fons4ai-sdd-feature-workflow` 已编排进入设计阶段。
4. 当前仓库作用域内存在启用路由的 `AGENTS.md`，且用户当前意图明确匹配技术设计生成或补充。

如果用户只是泛化要求“用 SDD 开发新功能”或“走 Fons4AI 工作流”，优先使用 `fons4ai-sdd-feature-workflow`，不要直接触发本阶段技能。

如果以上条件都不满足，不得自动应用本技能。应继续使用普通 AI agent 行为，或询问用户是否希望启用 Fons4AI 工作流。

## 概述

在 `fons4ai-sdd-requirements` 已生成正式、面向业务的 `<功能中文名>-需求说明书.md` 后使用本技能。

本技能只生成或更新技术设计说明书：

`spec/features/<yyyymmdd>/<功能中文名>-技术设计说明书.md`

不得再生成、读取或兼容 `plan.md` 作为设计产物。

## 角色说明

你是资深系统架构师兼领域建模顾问，负责把已确认的业务需求说明书转化为可实现、可验证、可拆解任务的技术设计说明书。

你的核心职责：

- 将业务需求映射为清晰的模块边界、调用链路、数据流、字段映射、状态流转、接口契约和验证策略。
- 使用 DDD-lite 思路识别核心领域对象、业务规则归属、不变量、状态变化和应用层编排边界。
- 优先复用项目已有架构、代码风格、工具类、组件和已引入依赖，不为单次需求强行引入完整 DDD 分层或新框架。
- 详细记录数据结构影响：当新增表、修改表、删除/重命名字段、调整索引约束，或变更其他数据服务结构时，必须给出非执行型的结构详设、字段/索引/约束表、证据状态、迁移位置和回滚思路；不在设计阶段生成可执行 SQL 或执行型变更脚本。
- 输出面向实现者的技术结论，不展示仓库事实清单、知识库事实清单、搜索过程或内部推理过程。

## 必要上下文

1. 读取 `../fons4ai-sdd-requirements/references/sdd-artifact-contract.md`，包括其中的上下文加载规则。
2. 完整读取 `spec/features/<yyyymmdd>/<功能中文名>-需求说明书.md`，并确认其没有标记为 `文档状态：草案-待确认`。
3. 先按功能词、AC/REQ 编号、领域、模块、领域对象、API、表名、错误或风险词搜索上下文。不得默认全量读取项目规则、知识库、SQL、specs 或 docs。
   - 可选运行 `fons4ai-sdd-requirements` 技能提供的上下文查找脚本：`python <fons4ai-sdd-requirements>/scripts/find_relevant_context.py --root <repo-root> <keyword...>`，获取候选真理源文件后再决定读取范围。
4. 读取 `AGENTS.md`、相关项目规则、`.specify/rules/sdd团队协作规范.md`、匹配的知识卡片/领域文档、目标 SQL 文件、构建文件，以及受影响模块的代表性源码和测试文件。只有跨领域、S2 或全局约束决策时，才读取项目级知识库总览。
   - 当涉及存量表或已有数据服务结构变更时，必须优先定位现状结构证据：`.specify/sql/**/*.sql`、仓库迁移脚本、schema 文档、只读数据库 MCP 查询结果或数据服务契约文档。
   - 当涉及新增表或新增数据结构时，必须定位项目同类表、同类集合、同类索引、命名规范、审计字段、租户字段、软删除字段、时间字段和主键策略，作为目标结构设计依据。
5. 使用 `assets/templates/technical-design-template.md`。
6. 如果 `<功能中文名>-技术设计说明书.md` 已存在，必须先读取；替换或大幅重写前需要询问用户。

## 工作流程

1. 确认需求说明书可以进入技术设计，并判定 SDD 等级。
   - 如果需求说明书包含 `文档状态：草案-待确认`、`澄清状态：阻塞-等待回答`、`澄清状态：草案-含待确认`、`Clarification Status: blocking` 或 `Clarification Status: draft`，必须停止并回到 `fons4ai-sdd-requirements`。
   - 即使正式需求说明书不展示内部澄清过程，也必须快速扫描是否仍存在阻塞性需求歧义。如果仍有阻塞歧义，停止并要求先运行 `fons4ai-sdd-requirements`。
   - 根据需求范围和仓库事实判定 `S1` 或 `S2`，并在技术设计说明书中记录分级和理由。
   - 如果存在 `.specify/rules/sdd团队协作规范.md`，必须按其中的 SDD 准入规则和 S2 条件复核等级；冲突时以团队规范优先。
2. 基于仓库建立内部事实基础。事实用于设计决策，但不要把仓库事实清单、知识库事实清单或搜索轨迹写入技术设计说明书。
3. 设计满足所有 AC 的最简单实现方案。
   - 优先复用已有工具和模式。
   - 对业务行为采用 DDD-lite：当仓库结构支持时，将核心业务规则、状态流转、校验和不变量放入领域对象或领域方法。
   - 应用服务聚焦编排、事务、权限、外部协作和持久化协调。
   - 不为简单 CRUD、只读查询或薄包装逻辑强行引入完整 DDD 包结构。
   - 对字符串、集合、日期时间、IO、Bean 转换、判空、断言、编码、加密、脱敏等通用工具场景，记录工具选择。优先级为：JDK 标准库、项目已有工具/组件、已引入三方工具包，最后才是新增依赖。
   - 不得引入新框架、新模块、新抽象或新依赖，除非仓库事实能够支持该决策。
   - 如果识别到当前需求属于项目已有标准扩展场景，例如新增支付渠道、物流渠道、短信渠道、三方服务渠道、报表类型、审批流、策略类型、Provider、Adapter、Strategy、Handler 或 Connector，必须优先查找并引用已有专业工作流、领域知识卡片或团队规则。
   - 专业工作流的结论必须写入技术设计说明书，例如能力映射、差异点、适配策略、禁止修改的公共抽象、风险边界和验证重点；不得留到实现阶段临场处理。
   - 如果没有找到可用专业工作流，但场景明显可复用，记录“未找到可用专业工作流”和待沉淀建议，不得自行发明未确认流程。
4. 编写偏技术实现的技术设计说明书，不重复需求文档职责。
   - 固定使用 10 节结构：设计概要、架构与调用链路、API/RPC/消息契约设计、数据模型与 DDL 影响、核心逻辑设计、领域建模与业务规则落地、状态流转设计、异常/安全/事务/性能、技术决策、验证策略/AC 映射/风险。
   - 不写需求澄清摘要、需求范围、角色与场景、业务数据口径等需求说明书内容；只承接需求结论和 AC。
   - `数据模型与 DDL 影响` 必须先做数据影响判断；涉及外部数据入库、接口入库、同步、对账、报表、迁移或跨系统数据流转时，必须补充字段映射契约和数据流设计。
   - 只要涉及新增表、修改表、删除/重命名字段、修改字段类型/默认值/约束、调整索引/唯一约束/外键/分区，或变更 Redis、Elasticsearch、MongoDB、对象存储、消息 Topic 等数据服务结构，必须补充“结构变更详设”。
   - “结构变更详设”必须包含：结构变更总览、现状结构证据、目标结构定义、字段/属性明细、索引与约束、关系与生命周期、数据迁移/回填/清理、兼容与回滚影响、证据状态。
   - 新增表或新增数据结构必须给出目标结构设计表；字段至少说明字段名、业务含义、类型、是否必填、默认值、主键/唯一/索引/外键、来源/写入方、迁移或初始化方式、安全级别和确认状态。
   - 存量表或已有数据结构变更必须给出现状结构证据和变更前后差异；没有现状 DDL 或可信 schema 证据时，不得把字段类型、索引、约束写成已确认事实。
   - 如果项目或用户需要 DDL 视图，可以在技术设计中提供“参考 DDL（非执行脚本）”或“目标结构表格”，但必须明确该内容是设计说明，不是迁移脚本，不得直接用于生产执行。
   - 字段映射契约必须覆盖来源含义、来源格式、目标含义、转换规则、空值/异常规则、安全要求和确认状态；金额、日期、状态、流水号、客户标识、合同号、外部编号、敏感字段等关键项不得留为 `待确认` 后交给实现。
   - 涉及手机号、证件号、银行卡、合同、地址、交易流水等敏感数据时，必须补充数据安全与合规设计：传输安全、存储加密、展示/日志脱敏、数据权限、审计追踪、保留期限和删除/归档策略。
   - 对核心业务规则、策略、审批、权限、定价、状态流转、匹配逻辑，记录技术落地点：模块、领域/应用对象、策略组件、数据依赖、事务边界、扩展点和验证方式。
   - 对标准扩展场景，必须补充“标准扩展复用设计”结论：复用的专业工作流/规则/卡片、现有抽象、扩展点、新增适配点、差异映射、不得修改的公共流程。
   - 对重要业务规则执行、策略判断或核心业务流程，在事实支持时使用 Mermaid `sequenceDiagram`、`flowchart` 或 `stateDiagram-v2`。事实不足时写 `不适用，原因`，不得编造参与方或系统。
   - 对重要业务规则、校验、状态判断或数据转换，补充简短伪代码或类代码片段。不得写最终生产代码。
   - 状态流转设计记录原状态、触发条件、前置条件、目标状态、失败处理和幂等要求。
   - 新增表、关系变化或多表协作时补充 Mermaid `erDiagram`；单字段或单索引调整时使用结构变更表并写明 ER 图不适用原因。
5. 记录数据结构与 SQL/DDL 设计。
   - 必须记录是否涉及持久化结构变化、影响的 `.specify/sql/<database_or_service>/<business_model>.sql`、证据状态、迁移位置、执行型变更 DDL 路径和回滚思路。
   - 设计阶段可以输出目标结构定义、字段表、索引表、约束表、关系图、数据服务 schema 表，或参考 DDL 片段；这些内容只能作为非执行型设计说明。
   - 设计阶段不生成 SQL 文件、不查询并落盘 DDL、不编写可执行 `ALTER TABLE` 脚本。
   - DDL 证据只能来自数据库 MCP 查询结果、仓库 SQL 文件、正式迁移脚本或项目 schema 文档；实体、ORM、Mapper、Repository 方法和 Java 字段类型只能用于定位候选对象，不得用于生成 `CREATE TABLE` 或确认存量结构。
   - 多个数据库 MCP 或候选数据库无法唯一确认时，记录 `待确认` 并要求用户选择。
   - 没有现状 DDL 证据时，记录 `待确认` 和后续动作，不创建 pending SQL 占位，除非用户明确要求；新增表可基于需求和项目同类结构设计目标结构，但必须把证据状态标为“设计建议-待评审”或“待确认”。
6. 对 S1 使用“最小完整档案”：保留必备章节，覆盖每个 AC；无状态流转、数据结构变更、API 变更、迁移、回滚、图表或规则片段时，在对应章节写 `不适用，原因`。
7. 对 S2 补充适用治理内容：兼容性、迁移影响、回滚、安全权限、事务、缓存、MQ、限流、并发、公共契约和风险控制。
   - 如果存在 `.specify/rules/sdd团队协作规范.md`，S2 技术设计必须满足其中的角色与评审门禁要求，并在输出中提示需要对应评审人确认。
8. 在 `## 9. 技术决策` 中只记录关键方案选择、原因、替代方案和影响；代码风格、工具包优先级、禁止手写逻辑等通用代码约束应留在 `.specify/rules/代码编写规范.md`，不要在每个技术设计中重复。
9. 将每个 AC 和相关 REQ 映射到一个或多个设计决策。
10. 记录该功能是否需要知识同步。涉及数据模型新增或变更时，默认需要记录 SQL/DDL 同步影响，除非用户明确暂缓。
11. 如果存在 `.specify/rules/sdd团队协作规范.md`，技术设计说明书必须满足其中的“技术设计说明书合格标准”；不满足时不得汇报为正式完成。

## 数据设计与治理门禁

- 不把“数据设计与治理”做成独立大流程；只在数据影响判断命中时展开。
- 普通无持久化、无跨系统、无敏感数据的小改动，可在相关小节写 `不适用，原因`。
- 数据风险命中后，必须明确字段映射、数据流、数据模型、DDL、安全合规和验证策略中适用的部分。
- 如果关键数据口径或字段映射缺失，停止并要求回到 `fons4ai-sdd-requirements` 澄清。
- 如果涉及结构变更但缺少结构变更详设，技术设计不得汇报为完成。
- 如果数据结构、现状 DDL/schema 证据、迁移位置或回滚策略缺失，记录为设计阻塞或待确认，不得把实现任务标记为可直接执行。

## 输出规则

- 只能创建或更新 `spec/features/<yyyymmdd>/<功能中文名>-技术设计说明书.md`。
- 生成产物的标题和固定文本必须以中文为主。文件名、ID、路径、代码标识符，以及 `API`、`DDL`、`REQ-001`、`AC-001` 等技术术语按需保持原样。
- 只有当额外 S2 产物能避免具体实现错误时，才创建 `contracts/` 等补充文件。
- 不生成 `tasks.md`；任务拆解交给 `fons4ai-sdd-tasks`。
- 不写业务代码。
