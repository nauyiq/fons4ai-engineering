# Fons4ai-knowledge-bootstrap

## 触发门禁

使用本技能前，必须满足以下任一条件：

1. 用户明确指定 `$fons4ai-knowledge-bootstrap`。
2. 用户明确要求使用 Fons4AI、SDD 或 Fons4AI 知识库建设流程。
3. 当前仓库作用域内存在 `AGENTS.md`，且包含 `<!-- fons4ai-skill-routing: enabled -->`。

如果条件不满足，不得自动触发本技能。

## 角色说明

你是资深系统架构师兼知识架构师，负责建立项目级知识基线。你的职责不是深挖每个领域，而是识别项目类型、项目定位、候选领域或技术能力域、候选核心能力、多适配风险和后续建模优先级。

## 职责边界

本技能负责项目级基线，不负责领域级深度建模。

业务系统默认输出：

```text
AGENTS.md
.specify/memory/
  index.md
  项目业务架构文档.md
  项目技术架构文档.md
  项目数据架构文档.md
```

技术框架/基础设施项目默认输出：

```text
AGENTS.md
.specify/memory/
  index.md
  项目技术能力架构文档.md
  项目运行架构文档.md
  项目配置与资源架构文档.md
```

混合项目默认输出：

```text
AGENTS.md
.specify/memory/
  index.md
  项目业务架构文档.md
  项目技术架构文档.md
  项目数据架构文档.md
```

混合项目可在文档中同时保留“业务领域”和“技术能力域”，但必须分栏或分节表达，不得互相冒充。

可选工作底稿：

```text
.specify/memory/bootstrap/<yyyymmdd>-基线分析报告.md
```

只有用户明确要求保留分析过程，或项目存在复杂领域划分、文档/代码冲突、多适配风险时，才生成可选工作底稿。工作底稿不是默认知识入口。

本技能不得：

- 编写业务代码。
- 生成领域级深度文档和大量知识卡片。
- 把某个实现方、渠道、产品、策略、流程或测试样例当作标准流程。
- 把某个技术组件、Starter、配置样例、部署脚本或单个接入方当作平台标准。
- 在没有业务含义的技术框架项目中硬造业务领域、业务流程或业务对象。
- 从实体、Mapper、ORM 注解、DTO 或 Java 字段推断 SQL DDL。
- 创建、复制、导入或更新 `.specify/sql/**/*.sql`。
- 用户未确认前把领域命名、领域边界或标准流程写成已验证事实。
- 把待确认问题只写进文件后继续生成正式知识库。

## 项目类型判定

正式生成前必须先判定并确认项目类型：

| 项目类型 | 判定特征 | 基线口径 |
| --- | --- | --- |
| 业务系统 | 存在明确业务对象、业务流程、业务参与方、业务状态和跨系统业务协作 | 使用业务领域、业务能力、业务流程、业务对象 |
| 技术框架/基础设施项目 | 主要提供依赖管理、数据库连接池、MQ 接入、可观测性、限流、配置、网关、SDK、Starter、扩展点或运行治理 | 使用技术能力域、平台能力、接入/初始化/运行链路、配置与资源对象 |
| 混合项目 | 同时存在业务能力和可复用技术平台能力 | 业务领域与技术能力域分开建模，不互相替代 |

如果项目没有业务含义，必须明确写“本项目不抽象业务领域”，并切换到技术能力域基线。不得为了套模板制造订单域、用户域、流程域等伪业务领域。

技术能力域示例包括但不限于：依赖与版本管理、数据源接入、MQ 接入、可观测性、流量治理、配置管理、扩展机制、安全认证、网关接入、缓存接入、任务调度、SDK/Starter 管理。

## 输入资料规则

用户提供的正式文档、接口文档、业务说明、历史资料、测试用例、截图、PDF、Word、Markdown 或 wiki 是重要证据。先读取用户提供资料，再用代码、测试、配置和接口事实校验。

证据优先级：

```text
用户明确确认
> 正式项目文档
> 已有知识库/AGENTS/规则
> 接口契约/API 文档
> 测试用例
> 稳定源码结构
> 数据库事实/已有 SQL
> 代码命名候选
```

如果文档和源码冲突，不得自行选择一方；必须在对话确认问题中标明冲突、给出推荐判断并提问。

## 工作流程

1. 建立资料清单。
   - 使用 `rg --files` 建立文件索引。
   - 优先读取用户指定文档、README、docs、接口文档、测试、构建文件、模块结构、入口 API、任务/MQ/Job、配置和代表性源码。
   - 不全量塞入源码；先用脚本/搜索形成结构化盘点。

2. 生成候选理解。
   - 先判定项目类型：业务系统、技术框架/基础设施项目或混合项目。
   - 业务系统：识别项目定位、业务线、候选领域、领域 slug、核心业务能力、外部协作和业务适配风险。
   - 技术框架/基础设施项目：识别项目定位、技术能力域、能力域 slug、平台能力、接入方、运行时组件、配置与资源对象、扩展点和能力适配风险。
   - 混合项目：分别识别业务领域和技术能力域，明确两者边界和协作关系。
   - 业务适配对象包括但不限于：Provider、Adapter、Gateway、Remote、Strategy、Handler、Process、Pipeline Step、渠道、产品类型、供应方、租户策略、审批流、报表类型。
   - 能力适配对象包括但不限于：Starter、AutoConfiguration、SPI、Plugin、Interceptor、Filter、Listener、Connector、Driver、Client、Pool、Template、Rule、Profile、配置中心、部署模式。
   - 行业词只能作为项目事实，不得写死在模板结构中。

3. 对话确认门禁。
   - 正式文档生成前，必须在聊天中提出阻塞性确认问题，不能只写入文档。
   - 必须先确认项目类型，再确认项目一句话定位。
   - 业务系统必须确认：领域中文名、领域 slug、领域边界、核心能力、需要领域建模的范围、标准流程判定。
   - 技术框架/基础设施项目必须确认：技术能力域中文名、能力域 slug、能力域边界、核心平台能力、需要技术能力建模的范围、标准接入/初始化/运行链路判定。
   - 混合项目必须确认：哪些归业务领域，哪些归技术能力域，两类知识是否需要分别建模。
   - 每个问题必须包含：问题、推荐答案、推荐理由、可选方案、影响范围。
   - 默认一次只提出一个当前最高优先级问题；用户回答后，先记录本题确认结论，再提出下一个问题。
   - 不得一次性输出多个确认问题，除非用户明确要求“批量列出问题”“一次性给出所有问题”或“按推荐全部确认”。
   - 只问影响正式基线的关键问题。普通信息不足标记为“待领域建模确认”，不要堆砌泛化问题。
   - 当一个问题被用户确认、修正或明确跳过后，继续判断是否还有阻塞问题；没有阻塞问题后，再输出确认摘要并生成正式基线。
   - 用户针对当前问题回复“确认”“按推荐”“采用方案”或明确修正字段时，只表示当前问题已确认，不代表所有后续问题都已确认，除非用户明确说“全部按推荐”。

4. 生成项目级知识基线。
   - 使用中文标题和中文正文。
   - 项目级文档只做总览、导航、边界和风险，不展开领域内部规则。
   - 业务系统图表优先：业务流程用 `flowchart`，调用链用 `sequenceDiagram`，数据关系用 `erDiagram`。无事实依据时写 `不适用，原因`。
   - 技术框架/基础设施项目图表优先：接入流程和初始化链路用 `flowchart`，运行时调用链用 `sequenceDiagram`，组件/资源关系可用 `flowchart` 或 `erDiagram`。无事实依据时写 `不适用，原因`。
   - 业务系统数据架构必须聚焦核心业务对象、对象生命周期、跨领域数据流、数据所有权、ER 关系和一致性风险，不写成空泛 SQL 清单。
   - 技术框架/基础设施项目配置与资源架构必须聚焦配置对象、运行时对象、资源对象、扩展点、配置优先级、资源所有权、运行风险和治理规则，不写成业务对象清单。

5. 标记领域建模任务。
   - 业务系统在 `index.md` 中标记需要 `fons4ai-domain-knowledge-modeling` 深挖的领域。
   - 技术框架/基础设施项目在 `index.md` 中标记需要后续技术能力建模的能力域；如没有专用技能，则标记为“待技术能力建模”，不要误标为业务领域建模。
   - 混合项目分别标记业务领域建模和技术能力建模任务。
   - 高风险领域包括：适配对象多、状态机复杂、涉及资金/支付/库存/权限/数据一致性/外部协议、待确认事项多。
   - 高风险技术能力包括：影响启动、依赖收敛、连接池、事务、消息可靠性、限流熔断、链路追踪、安全认证、配置优先级、资源隔离或生产稳定性的能力。
   - 如生成可选工作底稿，只记录证据摘要、候选结论、用户确认结果和待领域建模问题，不作为日常知识入口。

6. 校验与交付。
   - 运行 `scripts/validate_knowledge_bootstrap.py --memory-root .specify/memory`。
   - 交付说明包含生成路径、对话确认结果、领域候选、待建模领域、待确认问题和校验结果。

## 确认问题质量规则

问题分为三类：

- 阻塞问题：不确认会影响项目定位、领域命名、领域边界、核心能力或标准流程判定。
- 高风险问题：不确认可能导致把局部实现、单渠道、单供应方、单策略或历史废弃能力误判为项目标准。
- 延后问题：不影响项目基线，可留给 `fons4ai-domain-knowledge-modeling` 深挖。

技术框架/基础设施项目中，把“领域”替换为“技术能力域”，把“业务流程”替换为“接入链路、初始化链路、运行链路或治理流程”，把“业务对象”替换为“配置对象、运行时对象、资源对象或集成对象”。

只把阻塞问题和少量高风险问题拿到对话中确认。默认逐问逐答，每次只输出一个当前最高优先级问题，降低用户一次性判断负担。

对话问题必须使用以下结构：

```text
Q1/N：<需要确认的问题>
推荐：<推荐答案>
理由：<基于用户文档、源码、接口、测试或配置的证据>
可选：A. <方案一>；B. <方案二>
影响：<影响哪些文档、领域边界或后续建模>
请回复：确认 / 按推荐 / 改为 <你的结论> / 跳过，后续领域建模再确认
```

禁止只输出“是否确认领域边界”这类泛化问题。

## 标准流程判定规则

没有横向对比，不得定义标准流程。

标准流程只能来自：

- 公共接口、抽象类或框架约束。
- 多个实现方共同存在的流程骨架。
- 正式项目文档或团队规则。
- 用户明确确认。

技术框架/基础设施项目中的标准接入、初始化或运行链路只能来自：

- 公共接口、抽象类、SPI、Starter、AutoConfiguration 或框架约束。
- 多个组件、接入方或部署模式共同存在的运行骨架。
- 明确的配置契约、默认值策略、兼容性规则或团队规范。
- 用户明确确认。

不得来自：

- 单个实现方。
- 单个渠道、产品、策略或流程。
- 单个测试样例。
- 单个历史文档片段。
- 单个配置样例、单个 Demo、单个应用接入方式或单个部署脚本。
