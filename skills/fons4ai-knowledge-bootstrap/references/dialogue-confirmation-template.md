# 对话确认问题模板

> 用途：在生成正式知识库基线前，直接在对话中向用户确认关键问题。
> 约束：不得只把问题写入文档后继续生成正式知识库。

## 输出规则

- 只提出会影响正式基线的关键问题。
- 默认逐问逐答，每次只输出一个当前最高优先级问题。
- 用户回答当前问题后，先记录确认结论，再提出下一个问题。
- 不得一次性输出多个确认问题，除非用户明确要求“批量列出问题”“一次性给出所有问题”或“全部按推荐”。
- 问题必须具体，不能只问“是否确认领域边界”。
- 用户针对当前问题回复“确认”“按推荐”“采用方案”或明确修正字段时，只代表当前问题已确认。
- 没有剩余阻塞问题后，输出确认摘要，再生成正式基线。
- 普通待补充信息写入“待领域建模确认”，不要阻塞项目级基线。

## 问题格式

```text
Q1/N：<需要确认的问题>
推荐：<推荐答案>
理由：<证据和架构判断>
可选：A. <方案一>；B. <方案二>
影响：<影响的文档、领域边界、数据流或后续建模>
请回复：确认 / 按推荐 / 改为 <你的结论> / 跳过，后续领域建模再确认
```

## 问题分级

| 类型 | 是否必须对话确认 | 判断标准 |
| --- | --- | --- |
| 阻塞问题 | 是 | 影响项目定位、领域命名、领域边界、核心能力、标准流程判定 |
| 高风险问题 | 是，数量要少 | 可能把单实现、历史废弃能力、单渠道或单策略误判为标准 |
| 延后问题 | 否 | 不影响项目基线，可交给领域建模深挖 |

## 推荐问题类型

| 问题类型 | 示例 |
| --- | --- |
| 项目定位 | “该项目一句话定位是否应表述为……？” |
| 领域命名 | “推荐叫订单域而不是订单履约域，是否确认？” |
| 领域边界 | “支付渠道接入归入支付域，还是独立为渠道管理域？” |
| 废弃能力 | “代码存在 PBC 模块，但资料显示已废弃，是否不纳入候选领域？” |
| 标准流程 | “当前只找到某一个供应方实现，是否允许作为代表性实现而非标准流程？” |
| 数据事实源 | “订单主状态以订单表为事实源，支付记录只作为协同结果，是否确认？” |
