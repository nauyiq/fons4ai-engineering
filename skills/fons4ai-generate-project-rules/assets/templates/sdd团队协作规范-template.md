# SDD团队协作规范

> 适用范围：<项目/团队/仓库>
> 规则状态：初稿 | 已评审 | 待补充
> 更新日期：YYYY-MM-DD

## 1. SDD使用准入规则

| 场景 | 处理方式 | SDD等级 | 说明 |
| --- | --- | --- | --- |
| 全新业务功能 | 必须走 SDD | S1/S2 | 先生成需求说明书，再生成技术设计说明书，再生成任务规划 |
| 已有功能业务变更 | 必须走 SDD Change | S1/S2 | 生成 CR 和增量任务 |
| 涉及核心状态流转 | 必须走 SDD | S2 | 必须说明状态、兼容、回滚和回归验证 |
| 涉及数据库结构变更 | 必须走 SDD | S2 | 必须说明 DDL 影响、迁移、回滚和验证方式 |
| 涉及文件入库、接口入库、数据同步、对账或报表 | 必须走 SDD | S1/S2 | 必须说明数据口径、字段映射、数据流和样例验证 |
| 涉及金额、日期、状态、流水号、客户标识等关键数据 | 必须走 SDD | S1/S2 | 关键字段口径未确认前不得实现 |
| 涉及敏感数据、隐私数据或监管数据 | 必须走 SDD | S2 | 必须说明脱敏、加密、权限、日志、审计和保留策略 |
| 涉及权限、安全、资金、风控 | 必须走 SDD | S2 | 必须经过人工 Gate 后才能实现或上线 |
| 普通 BUG 修复 | 走 bugfix workflow | 不适用 | 先复现、定位、修复、回归 |
| 文案、样式、小配置、日志级别 | 走 S0 快路径 | S0 | 生成快速变更记录，确认后实现，验证后回写结果 |
| 依赖版本升级（无破坏性变更） | 走 S0 快路径 | S0 | 生成快速变更记录，确认后实现，验证后回写结果 |
| 纯代码重构（不改变行为） | 走 S0 快路径 | S0 | 生成快速变更记录，确认后实现，验证后回写结果 |
| 纯代码重构（影响公共行为） | 必须走 SDD | S1/S2 | 不改变业务行为时可走 S0；影响公共行为时必须走 SDD |

## 2. SDD产物质量标准

### 2.1 需求说明书合格标准

- 必须说明业务背景、目标和范围。
- 必须说明核心角色、业务场景和业务流程。
- 必须说明核心业务规则、业务数据口径和业务影响。
- 涉及数据存储、流转、字段映射、金额、日期、状态、流水号、客户标识或敏感数据时，必须包含数据影响判断和关键数据口径。
- 必须包含可测试的验收标准。
- 必须明确待确认事项。
- 不得把技术实现、类名、表名、字段名、DDL 路径写入需求说明书。

### 2.2 技术设计说明书合格标准

- 必须说明总体实现方案。
- 必须说明核心业务规则如何落地。
- 必须说明接口、数据流、状态流转、事务一致性和异常处理。
- 涉及入库、出库、同步、对账、报表或跨系统数据流转时，必须说明字段映射契约、数据流设计和样例验证策略。
- 涉及数据结构变更时，必须说明数据模型、DDL 影响、迁移和回滚。
- 涉及敏感数据时，必须说明脱敏、加密、权限、日志、审计、保留期限和删除/归档策略中适用的部分。
- S2 必须说明兼容性、安全、回滚和风险控制。
- 不得重复堆砌需求说明书内容。

### 2.3 任务规划合格标准

- 每个任务必须可独立执行。
- 每个任务必须包含 `AC`、`Files`、`Verification`、`Quality`、`Done`。
- 必须覆盖开发、测试、数据、回归和风险控制任务。
- 涉及数据风险时，必须包含字段映射验证、样例数据验证、数据安全检查或数据一致性验证任务。
- 涉及用户手动执行 DDL 时，必须保留执行前置条件和执行后验证任务。
- 任务规划必须标记高风险项和人工 Gate。
- 生成任务规划后必须暂停，等待用户确认执行。

## 3. 角色与评审门禁

| 产物或场景 | 主要评审人 | 评审重点 |
| --- | --- | --- |
| 需求说明书 | 产品经理 / 业务负责人 | 业务目标、范围、规则、验收标准 |
| 技术设计说明书 | 技术负责人 / 架构师 | 架构方案、领域边界、接口、数据、风险 |
| 任务规划 | 技术负责人 / 开发负责人 | 任务拆解、顺序、可验证性、遗漏风险 |
| 实现产物 | Spec Reviewer | 是否符合需求、设计、任务 AC，是否漏做或多做 |
| 实现产物 | Code Reviewer | 代码质量、边界条件、异常处理、安全、兼容性和测试 |
| S2 变更 | 架构师 / 技术负责人 / 测试负责人 | 兼容、回滚、迁移、安全、回归 |
| 数据结构变更 | DBA / 后端负责人 | DDL、数据兼容、迁移、回滚 |
| 权限、安全、资金、风控 | 安全负责人 / 业务负责人 / 技术负责人 | 权限边界、安全风险、资金正确性 |

实现阶段必须遵守以下分权门禁：

- 实现 Agent（Implementer）负责编码和本地验证，不得直接裁决完成。
- Spec Reviewer 必须先于 Code Reviewer 进行，先确认做对需求，再确认代码质量。
- Code Reviewer 不得替代 Spec Reviewer；测试通过也不得替代需求符合性检查。
- Reviewer 不应继承 Implementer 的完整上下文，只读取需求、设计、任务、diff 和验证证据。
- Reviewer 发现 Critical 或 Important 问题时，必须修复、复测、复审。
- 高风险项未通过人工 Gate 时，只能声明“实现候选完成”，不得声明“可交付完成”。

## 4. 评审结论

评审结论必须是以下之一：

- `通过`
- `有条件通过`
- `退回修改`
- `暂缓`

`有条件通过` 必须明确条件、责任人和关闭方式。
`退回修改` 必须明确阻塞项、证据和建议修复方向。`暂缓` 必须说明缺失信息、外部依赖或风险原因。

## 5. 标准扩展场景与专业工作流复用

当需求属于项目已有能力的标准化扩展场景时，必须优先复用已有专业工作流、领域知识卡片或团队规则，而不是让 Agent 自由发挥。

典型标准扩展场景包括：

- 新增支付渠道、物流渠道、短信渠道、三方服务渠道。
- 新增报表、审批流、业务策略、风控规则或营销活动类型。
- 在已有抽象下新增 Provider、Adapter、Strategy、Handler、Job、Connector。

复用规则：

- 需求或变更阶段必须识别是否属于标准扩展场景。
- 技术设计阶段必须吸收专业工作流结论，例如差异映射、接入策略、风险边界和禁止修改的公共抽象。
- 任务规划阶段必须把专业工作流约束拆入 `Quality`、`Verification` 或 `Done`，不得只写“参考某工作流”。
- 实现阶段只能读取技术设计或任务规划已明确引用的专业工作流、规则或知识卡片。
- 实现阶段不得因为存在专业工作流就新增未规划任务、修改公共抽象、改变 AC 或扩大技术范围。

## 6. 标准样例库

团队成员和 Agent 生成 SDD 产物时，应优先参考：

- `.specify/examples/sdd/s1-simple-feature/`
- `.specify/examples/sdd/s2-risk-feature/`
- `.specify/examples/sdd/change-feature/`

样例只作为格式和质量参考，不得复制其中业务内容到真实项目。没有样例库时，不阻塞 SDD 执行，但应按本规范的质量标准生成产物。

## 7. SDD完成定义

一个 SDD 需求只有同时满足以下条件，才视为完成：

S0 快速变更记录的完成标准：

- 快速变更记录已生成并通过 S0 校验。
- 用户确认实现后变更已完成。
- 验证结果已回写快速变更记录。
- 变更文件清单与实际修改一致。
- 验证失败时保留未完成状态，不标记完成。

S1/S2 SDD 需求的完成标准：

- 需求说明书、技术设计说明书、任务规划均已生成或更新。
- 需求、技术方案和任务规划已按团队规则完成必要评审。
- 所有确认执行的任务已完成。
- 自动化测试或必要手动验证已完成，并记录结果。
- Spec Review 已确认实现符合需求、设计和任务 AC。
- Code Review 已确认代码质量、异常处理、安全、兼容性和测试证据可接受。
- 涉及高风险项时，人工 Gate 已通过或明确记录未通过原因。
- 涉及数据结构变更时，DDL 影响、执行方式和验证结果已记录。
- 涉及接口、业务规则、领域边界、数据模型或长期技术事实时，已判断是否需要知识汇总。
- 若需要知识汇总，已通过 `fons4ai-knowledge-summary` 更新知识库，或明确记录未同步原因。
- 交付说明已记录变更内容、涉及文件、验证结果、风险和后续事项。

最终交付必须附 Evidence Bundle，至少包含任务来源、变更范围、验证命令、测试说明、Review 状态、人工 Gate 状态、风险声明和未验证项。
