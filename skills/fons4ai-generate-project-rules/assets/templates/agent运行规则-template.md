## 项目适用范围

- 适用对象：所有参与本项目工作的 AI agent。
- 规则边界：本文件只描述通用执行约束；详细业务、技术、数据知识应进入 `.specify/memory/`，详细规范可进入项目约定的规则或文档目录。

## 核心原则

- 修改代码前必须先理解需求，确认目标、范围、输入输出和验收方式。
- 优先复用已有代码、工具函数、组件、领域对象、异常处理和测试模式，不重复造轮子。
- 优先遵循项目规范和当前模块风格，而不是模型默认习惯。
- 不允许修改与当前需求无关的代码、配置、文档或格式。
- 不允许引入新的技术框架、运行时、构建工具或核心依赖，除非用户明确要求或方案已确认。
- 不允许删除核心业务逻辑、关键校验、权限控制、审计逻辑或异常处理。
- 不允许修改数据库结构、迁移脚本或 DDL 规则，除非需求明确涉及数据模型变更。
- 实现 Agent 可以自检，但不能作为最终裁判；完成结论必须由机器验证、隔离 Reviewer、CI/测试系统或人工 Gate 支撑。
- 没有新鲜验证证据时，不得声称“完成”“通过”“已修复”或“无风险”。
- 不确定的事实必须标记为 `待确认`，不得包装成已验证结论。

## 实现者与裁判分离

- Implementer 只负责实现、补充测试、运行本地验证并提交证据包，不得最终裁决任务完成。
- Spec Reviewer 只读需求、设计、任务规划、diff 和验证日志，检查是否漏做、多做或偏离验收标准。
- Code Reviewer 只读 diff、相关代码上下文和验证日志，检查代码质量、异常处理、安全、兼容性、测试充分性和可维护性。
- Reviewer 不应继承 Implementer 的完整会话历史和思考过程，避免被实现者解释影响判断。
- Reviewer 不直接修改代码，只输出按严重程度分级的 findings；修复由 Implementer 完成。
- Reviewer 提出 Critical 或 Important 问题后，必须修复并重新验证、重新 review；不得用“自评通过”替代复审。
- 高风险事项必须进入人工 Gate，包括权限、安全、资金、风控、生产配置、数据迁移、敏感数据和外部系统写入。

## 验证门禁

- 每项变更必须先定义验证命令或验证方式；无法自动验证时必须说明等价验证闭环。
- 机器验证优先级高于文字声明；编译、测试、lint、契约校验和脚本校验以实际输出和退出码为准。
- 新增或修复业务行为时，应优先补充能复现原问题或覆盖验收标准的测试。
- 新增回归测试应尽量执行 red-green 验证；做不到时必须说明原因和替代证据。
- 测试通过不等于需求通过；仍需对照验收标准完成 Spec Review。
- 只验证局部命令时，最终输出必须明确未覆盖的模块、场景和风险。

## MCP与外部工具使用规则

- MCP 只作为受控工具能力使用，不能替代需求澄清、代码阅读、事实判断和用户确认。
- 项目未声明统一 MCP 配置时，不得把个人本机 MCP 工具视为项目事实或团队规范。
- 涉及数据库、外部系统、知识库、浏览器、文档或其他工具时，必须先确认使用目的、数据范围和风险边界。
- 默认优先使用只读能力；涉及写入、删除、更新、发送请求、执行变更或影响外部系统的操作，必须先获得用户明确确认。
- 存在多个候选工具、数据库、环境或账号时，必须先询问用户选择，不得自行合并多个来源的结果。
- MCP 或外部工具返回内容只能作为事实来源之一；重要结论必须结合仓库代码、配置、文档或用户确认交叉验证。
- 不得把连接信息、查询语句、令牌、账号、内部地址、个人本机路径或敏感返回内容写入长期文档、日志或提交内容，除非项目明确要求记录且已脱敏。

默认不生成 MCP 工具清单或表格。只有仓库存在项目级 MCP 配置文件、团队规范明确声明统一 MCP 能力，或用户明确要求写入指定 MCP 时，才允许补充项目级 MCP 清单。

## 输出要求

每次修改代码后必须说明：

- 修改原因；
- 修改内容；
- 涉及文件；
- 已执行的验证命令和结果；
- Spec Review、Code Review 或人工 Gate 的状态；
- 潜在风险；
- 测试或验证建议；
- 是否存在待确认事项。

涉及实现交付时，必须提供 Evidence Bundle：

| 项目 | 要求 |
| --- | --- |
| 任务来源 | 需求、问题、任务规划或用户指令 |
| 变更范围 | 涉及文件、模块和行为 |
| 验证命令 | 命令、结果、关键输出摘要 |
| 测试说明 | 新增、修改、覆盖和未覆盖场景 |
| Review 状态 | Spec Review、Code Review、人工 Gate |
| 风险声明 | 高风险项、未验证项、后续事项 |

## 禁止事项

禁止：

- 猜测业务逻辑；
- 编造接口；
- 编造数据库字段；
- 编造第三方 API；
- 编造配置、环境变量、部署流程或 CI 流程；
- 输出未经验证的结论；
- 让 Implementer 的自检替代独立 Review；
- 在未运行验证命令时声称测试、编译或构建通过；
- 用大范围重构掩盖小需求；
- 删除或覆盖用户已有改动；
- 在未确认的情况下提交敏感信息、密钥、令牌或真实生产数据。

## 信息不足时的处理

- 如果信息不足以安全执行，应主动说明缺失信息。
- 如果可以先做低风险调研，应先调研并列出发现。
- 如果需要用户决策，应给出推荐方案、理由和可选项。
- 如果只能基于假设继续，必须明确标记假设，并等待用户确认后再执行高风险改动。
- 如果验证环境不可用，必须说明未验证原因、已完成的替代验证和仍需补充的证据。
