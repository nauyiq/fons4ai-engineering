# 代码编写规范

> 适用范围：本项目所有由 AI agent 参与编写或修改的代码
> 生成依据：用户约定、项目知识库、已有代码风格、当前模块上下文
> 规则状态：初始约定 | 已评审 | 待调整

## 基本原则

- 编写代码前必须先阅读当前需求、相关源码、相关测试和必要的项目知识库。
- 新代码必须优先贴合当前模块已有风格，不得使用模型默认习惯替代项目风格。
- 代码改动必须聚焦当前需求，不得夹带无关重构、格式化、重命名或目录调整。
- 不得为了“看起来更完整”新增未被需求要求的功能、接口、字段、配置或扩展点。
- 不确定的业务规则、接口语义、字段含义、错误码、权限规则必须标记为 `待确认`，不得自行补全。

## 工具类与复用

- 编写通用逻辑前必须先搜索已有工具函数/工具类、基础组件、公共方法和相邻模块实现。
- 工具能力优先级为：当前语言标准库 -> 项目已有工具/基础组件 -> 项目已引入三方工具库 -> 新增依赖。
- 字符串、集合、日期时间、金额、IO、对象/数据转换、判空、断言、加解密、脱敏等通用逻辑，禁止重复手写低质量实现。
- 如果项目已引入对应语言生态的工具库，应优先复用其成熟方法；例如 JVM 的 Hutool、Apache Commons、Guava 或 JavaScript 的 Lodash，均以仓库事实为准。
- 新增工具方法必须满足：当前项目没有等价能力、方法职责单一、命名表达用途、边界条件清晰、有测试或明确验证方式。
- 新增三方依赖前必须获得设计方案或用户确认，不得直接修改依赖文件。

## 代码风格

- 命名必须表达业务含义，避免 `data`、`temp`、`obj`、`handle`、`process` 等含糊名称。
- 方法应保持单一职责；一个方法不应同时承担校验、查询、转换、持久化、外部调用和响应组装。
- 条件复杂时，应提取具名变量、私有方法、策略对象或领域方法。
- 不得复制粘贴大段相似代码后只改少数字段；出现重复规则时应优先抽取或复用已有能力。
- 不得为了少写代码牺牲可读性；过长链式调用、过深嵌套、复杂 Lambda 应拆分。
- 新增关键业务逻辑、复杂分支、兼容处理、领域字段含义时，应添加简洁注释说明意图。

## DDD-lite 编码约束

- 核心业务规则、状态流转、校验和不变量应优先放入领域对象、领域方法或领域服务。
- 应用服务优先负责用例编排、事务、权限、外部协作和持久化协调。
- Controller、RPC 接口、Adapter 不应承载复杂业务规则。
- Mapper、Repository、DAO 不应承载业务决策。
- 简单 CRUD、纯查询、DTO/VO、查询投影、兼容旧结构可以保持轻量模型。
- 如果核心业务逻辑暂时留在应用层，必须说明原因，例如项目无领域层、逻辑很薄、规则尚不稳定或兼容旧结构。

## API 接口设计

- 新增或修改 Controller、RPC、开放接口、消息消费、Webhook、第三方回调前，必须先确认业务语义和调用方。
- 接口命名、路径、方法、参数结构、响应结构必须遵循项目已有风格。
- 不得编造接口字段、响应字段、错误码、权限规则或第三方 API。
- 请求参数必须区分必填字段、可选字段、默认值、业务约束和兼容字段。
- 响应结果必须区分成功结果、业务失败、参数错误、权限失败和系统异常。
- 接口必须考虑幂等、重复提交、权限校验、参数校验和敏感信息脱敏。
- 对外接口变更必须考虑兼容性；删除字段、改字段含义、改错误码、改状态枚举前必须确认影响范围。
- 内部接口不应暴露数据库细节、临时实现细节或不稳定领域对象。

## 异常与日志

- 异常类型、错误码、返回格式必须沿用项目已有约定。
- 参数错误、业务失败、外部调用失败、数据异常应使用不同处理方式，不得统一吞掉。
- 不得吞异常；确需降级或忽略异常时，必须说明原因并保留必要日志。
- 日志必须能帮助定位问题，但不得输出密钥、令牌、手机号、身份证号、生产数据或完整敏感报文。
- 外部调用日志应记录调用目标、业务标识、结果状态和耗时，但必须避免敏感信息泄露。

## 数据访问与事务

- 数据访问必须使用项目已有模式，例如 Mapper、Repository、DAO、ORM Client 等。
- 不得在 Controller、RPC 接口、页面入口或 UI 层直接访问数据库。
- 事务边界应放在应用层或项目已有事务承载位置。
- 跨服务、跨库、外部调用、消息发送不得随意放入长事务。
- 状态更新必须考虑并发、幂等、重复请求和失败重试。
- 修改数据库结构、字段、索引、约束、默认值前，必须确认需求明确涉及数据模型变更，并按项目 DDL 规则处理。

## 数据映射与安全

- 文件入库、接口入库、数据同步、对账、报表和迁移代码必须先确认字段映射契约，不得凭字段名相似度或历史经验自行映射。
- 金额字段必须确认单位、精度、舍入规则和是否需要分/元转换；复用工具类前必须确认工具语义不会改变单位或精度。
- 日期时间字段必须确认格式、时区、业务日期和存储口径，不得混用字符串日期、业务日和系统时间。
- 状态、枚举、错误码、外部流水号、客户标识、合同号等关键字段必须确认来源和含义，不得用相邻字段临时代替。
- 样例数据验证应覆盖正常值、空值、异常值、边界金额、特殊日期、未知状态和外部编号缺失。
- 涉及手机号、证件号、银行卡、合同、地址、交易流水等敏感数据时，必须检查存储、展示、接口响应、日志、异常信息和测试数据是否符合脱敏、加密、权限和审计要求。
- 日志不得输出完整敏感字段、密钥、令牌、生产报文或可复原个人信息；必要时只输出业务追踪 ID、摘要或脱敏值。

## 测试与验证

- 行为变更应优先补充自动化测试。
- BUG 修复应优先补充能复现问题的测试或最小验证信号。
- 测试应复用已有测试夹具、构造器、Mock、Stub、Fake 和断言风格。
- 不得依赖真实生产服务、真实账号、真实敏感数据或不可重复环境。
- 无法自动化验证时，必须说明原因，并给出手动验证步骤和预期结果。

## 禁止事项

禁止：

- 未搜索已有实现就新增重复逻辑。
- 重复手写项目已有工具类或成熟三方库已覆盖的能力。
- 擅自新增三方依赖、框架、模块或公共抽象。
- 编造业务规则、接口、数据库字段、第三方 API、配置项或错误码。
- 在接口层堆积复杂业务规则。
- 在数据访问层编写业务决策。
- 删除核心业务逻辑、关键校验、权限控制、审计逻辑或异常处理。
- 修改与当前需求无关的代码、格式、命名或目录结构。
- 把临时调试、未验证假设或敏感信息写入代码、测试或文档。
