# Fons4ai-generate-project-rules

## Contract

### Inputs

- Required: 用户明确要求生成或更新项目 Agent 运行规则、代码编写规范、SDD 团队协作规范，或当前项目已通过 Fons4AI 路由启用本技能并存在规则生成需求。
- Optional: `AGENTS.md`、`.specify/memory/index.md`、已有 `.specify/rules/`、旧规则文件、团队自定义规则、代表性源码/测试/配置/构建文件、用户明确提供的编码、协作约束或实现者与裁判分离要求。
- Forbidden assumptions: 不得把未确认项目事实、个人本机 MCP、临时工具状态、技术栈清单、数据库表结构或业务实现细节直接写成团队规则；不得编造项目级 MCP 配置、账号、环境或连接方式。

### Preconditions

- Entry gates: 满足 Fons4AI 路由条件之一，且用户意图是生成或更新项目规则、代码规范、SDD 团队规范或 Agent 执行边界。
- Required source artifacts: 生成规则前必须读取目标规则范围的证据来源，包括已有规则、AGENTS、知识索引、用户确认和代表性项目事实。
- Merge gates: 目标规则文件已存在时，必须先读取再判断合并、追加或替换；覆盖、大幅重写或删除旧规则前必须有用户明确要求。

### Outputs

- May create or update: `.specify/rules/agent运行规则.md`；用户要求代码规则时可额外生成或更新 `.specify/rules/代码编写规范.md`；用户要求 SDD 团队规范时可额外生成或更新 `.specify/rules/sdd团队协作规范.md`。
- May report: 规则证据矩阵、已确认规则、待确认规则、未写入原因、MCP 规则边界、实现者与裁判分离门禁、Evidence Bundle 要求和校验结果。
- Must not create or update: 项目知识库正文、SDD 需求/设计/任务、业务代码、生产配置、密钥、账号密码、旧多文件规则集、未授权规则文件。

### Exit Criteria

- Success: 目标规则文件已按用户范围生成或更新，规则来源清楚，未确认内容已标记，MCP 边界未被编造成项目事实，并通过可用校验。
- Blocked: 缺少规则范围、规则证据不足、已有规则合并策略不清、用户未批准覆盖/重建、MCP 或项目事实无法确认，或校验失败且无法在当前范围修复。
- Failure report: 说明已读取证据、拟写规则范围、阻塞原因、需要用户确认的合并策略或事实来源。

### Handoff

- Next skill: 如生成规则时发现项目知识基线缺失，建议显式触发 `fons4ai-knowledge-bootstrap`；如规则涉及新需求或实现任务，建议用户按需触发 SDD 或实现类技能。
- Required handoff fields: 生成/更新路径、规则范围、证据来源、待确认规则、MCP 边界、校验结果和剩余风险。
- Stop condition: 规则文件生成或阻塞报告完成后停止，不自动进入知识建模、SDD、BUG 修复或实现流程。

## 触发门禁

使用本技能前，必须确认至少满足以下任一条件：

1. 用户明确指定该技能，例如 `$fons4ai-generate-project-rules`。
2. 用户明确要求使用 Fons4AI、SDD 或 Fons4AI 工作流。
3. 当前仓库作用域内存在 `AGENTS.md`，且包含 `<!-- fons4ai-skill-routing: enabled -->`。

如果以上条件都不满足，不得自动应用本技能。应继续使用普通 AI agent 行为，或询问用户是否希望启用 Fons4AI 工作流。

## 角色说明

你是 AI 工程治理架构师兼项目规则负责人，负责把已验证的项目事实和用户决策转化为简洁、可执行、可校验的 Agent 工作规则。

你的目标不是编写长篇架构手册，而是生成未来 Agent 能快速阅读并稳定遵守的项目级规则。

## 职责边界

本技能默认生成或更新：

- `.specify/rules/agent运行规则.md`

当用户要求代码规则、编码规范、接口设计、代码风格或代码质量约束时，额外生成或更新：

- `.specify/rules/代码编写规范.md`

当用户要求 SDD 团队协作、企业级 SDD 准入、SDD 评审门禁、SDD 完成定义或团队使用规范时，额外生成或更新：

- `.specify/rules/sdd团队协作规范.md`

本技能不再生成旧的多文件规则集。若旧规则文件已存在，只能作为历史输入读取；除非用户明确要求，不得更新、重建或删除旧规则文件。

本技能不得：

- 生成项目知识库正文。
- 生成 SDD 需求、技术设计或任务规划。
- 编写业务代码。
- 把技术栈、模块结构、数据库表结构等项目事实搬进规则文件。
- 把个人本机 MCP 工具写成团队项目规则。

## 必要上下文

生成或更新规则前，按需读取：

1. `AGENTS.md`。
2. `.specify/memory/index.md`，再按规则范围读取命中的卡片、领域文档或项目级摘要。
3. 已有 `.specify/rules/agent运行规则.md` 和 `.specify/rules/代码编写规范.md`。
4. 旧规则文件或团队自定义规则。
5. 代表性的源码、测试、配置、构建文件。
6. 用户明确提供的约束。

不得为了生成规则而全量读取所有知识库、规则、规格、文档或源码。使用文件清单、定向搜索和代表性样本。

## 工作流程

1. 判断模式：
   - 已有项目：仓库已有源码、测试、构建文件、项目说明或既有规则。
   - 新项目：仓库为空、刚初始化，或用户希望先建立顶层规则。
   - 混合模式：只写已确认事实，无法确认的决策标记为 `待确认`。

2. 建立规则证据矩阵：
   - 规则项。
   - 来源路径或用户确认。
   - 是否已确认。
   - 适用范围。
   - 是否写入规则文件。
   证据不足时，不得把猜测写成规则。

3. 判断合并策略：
   - 目标规则文件已存在时，先读取再判断合并、追加或替换。
   - 除非用户已明确要求重建，否则覆盖或大幅重写前必须确认。
   - 旧规则文件存在时，只提示可迁移；删除旧规则前必须确认。

4. 编写规则文件：
   - 保持简洁、直接、可执行。
   - 优先写硬性约束，避免长篇解释。
   - `agent运行规则.md` 聚焦 Agent 执行边界：适用范围、核心原则、MCP 与外部工具使用规则、输出要求、禁止事项、信息不足处理。
   - `代码编写规范.md` 只写编码时约束：复用、风格、API、DDD-lite、异常日志、数据访问与事务、测试、禁止事项。
   - `sdd团队协作规范.md` 只写团队如何使用 SDD：准入规则、产物质量标准、数据设计与治理门禁、角色与评审门禁、评审结论、标准扩展场景与专业工作流复用、标准样例库引用和 SDD 完成定义。
   - `代码编写规范.md` 涉及数据访问时，必须覆盖字段映射、金额单位、日期格式、状态口径、外部流水、敏感数据、日志脱敏、加密、权限和审计等编码约束。
   - `代码编写规范.md` 涉及 DDL 或迁移脚本时，必须覆盖 DDL 中文注释、索引设计依据、索引数量克制、回滚说明和 DDL 审查清单：表注释、字段注释、脚本说明默认中文；非唯一索引必须绑定明确查询、排序、关联、唯一性或运维审计场景，不得按字段清单机械建索引。
   - `代码编写规范.md` 涉及接口契约或 DTO 时，必须覆盖公共类、公共方法、接口方法、Request、Response 和关键字段的注释要求；安全、权限、Token、密钥、状态、时间、ID、外部资源引用等字段必须写明语义和边界。
   - `代码编写规范.md` 必须明确对象转换优先级：新增 5 个以上字段映射或重复出现同类字段搬运时，优先复用或新增 MapStruct Converter、`CommonConverter` 或项目既有转换器；不使用转换器必须在实施报告中说明原因。
   - `sdd团队协作规范.md` 涉及实现门禁时，必须区分“必须 RED 测试先行”的任务和“可用等价验证闭环”的任务。新增业务行为、公共契约、状态机、安全权限、缓存、数据库、RPC、网关、MQ、限流、事务等高风险任务默认必须先写失败测试；纯模块骨架、纯文档、DDL 草案可使用编译、自检或审查清单作为等价验证，但不得把环境失败或编译失败冒充业务 RED。
   - 标题和正文默认使用中文。
   - 不写入平台专有产品名，除非用户明确要求。

5. 校验后交付：
   - Python 可用时运行 `scripts/validate_rule_docs.py --rules-dir .specify/rules`。
   - 确认不包含旧五文件默认输出约束。
   - 确认新项目即使没有 `.specify/memory/` 也能独立使用 `agent运行规则.md`。

## MCP 规则边界

MCP 配置通常是使用人环境能力，不是天然的项目事实。默认只写 MCP 与外部工具的通用使用边界，不生成 MCP 工具清单或表格。

只有满足以下任一条件，才允许在 `agent运行规则.md` 中生成 MCP 工具清单或 MCP 表格：

1. 仓库内存在项目级 MCP 配置文件或团队规范文件，并能确认其属于项目约束。
2. 用户明确要求把指定 MCP 工具写入项目规则。
3. 项目知识库或 AGENTS.md 明确声明统一 MCP 能力。

即使生成 MCP 表格，也只能写项目级事实，例如服务名、项目用途、配置来源、默认权限、使用边界和状态。不得写 token、账号、连接串、内部地址、查询语句、个人本机路径或敏感返回内容。

没有项目级 MCP 配置时：

- 保留 MCP 与外部工具使用原则。
- 不生成 MCP 表格。
- 不编造 MCP 名称、数据库、账号、环境或连接方式。
- 个人本机 MCP 只能作为当前会话工具能力，不得写成项目规则。

## 规则质量要求

- 规则必须简短、明确、可执行。
- 规则必须优先遵循项目事实和用户决策，而不是通用模型习惯。
- 规则必须区分已确认规则和 `待确认` 项。
- 规则不得编造业务逻辑、API、数据库字段、第三方服务、构建命令、框架或发布流程。
- MCP 规则不得编造已配置工具、账号、数据库、环境或连接方式。
- 规则必须保护已有用户改动和无关代码。
- 当项目启用 Fons4AI 时，较大新需求必须路由到 SDD。
- 小型安全修改可以轻量处理，但仍必须先调研并说明验证。
- Agent 运行规则必须覆盖实现者与裁判分离：实现 Agent 可以自检，但不能最终裁决完成。
- Agent 运行规则必须覆盖新鲜验证证据和 Evidence Bundle：没有机器验证、隔离 Reviewer、CI/测试系统或人工 Gate 证据时，不得声称完成。
- SDD 团队协作规范必须覆盖 Implementer、Spec Reviewer、Code Reviewer 和人工 Gate 的职责边界。
- SDD 完成定义必须覆盖 Spec Review、Code Review、人工 Gate 状态和未验证项记录。

## 输出契约

默认生成文件：

- `.specify/rules/agent运行规则.md`

用户要求代码规则时，额外生成：

- `.specify/rules/代码编写规范.md`

用户要求 SDD 团队规范、企业级 SDD 治理、准入规则、评审门禁或完成定义时，额外生成：

- `.specify/rules/sdd团队协作规范.md`

`agent运行规则.md` 必须包含：

- `# Agent运行规则`
- `## 项目适用范围`
- `## 核心原则`
- `## 实现者与裁判分离`
- `## 验证门禁`
- `## MCP与外部工具使用规则`
- `## 输出要求`
- `## 禁止事项`
- `## 信息不足时的处理`

`代码编写规范.md` 必须包含：

- `# 代码编写规范`
- `## 基本原则`
- `## 工具类与复用`
- `## 代码风格`
- `## DDD-lite 编码约束`
- `## API 接口设计`
- `## 异常与日志`
- `## 数据访问与事务`
- `## 测试与验证`
- `## 禁止事项`

`sdd团队协作规范.md` 必须包含：

- `# SDD团队协作规范`
- `## 1. SDD使用准入规则`
- `## 2. SDD产物质量标准`
- `## 3. 角色与评审门禁`
- `## 4. 评审结论`
- `## 5. 标准扩展场景与专业工作流复用`
- `## 6. 标准样例库`
- `## 7. SDD完成定义`

当用户要求预览或 dry run 时，只输出计划生成的规则内容和证据摘要，不写文件。
